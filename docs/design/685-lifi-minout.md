## 1. Verified premise (what the code actually does)

DEV's #722 claim is **CORRECT in substance, with one citation error**.

- **The MCP forwards LiFi's calldata verbatim and does not own `_minAmountOut`.** `prepareSwap` reads `quote.transactionRequest` and copies `to` / `data` / `value` straight into the returned `UnsignedTx` — `src/modules/swap/index.ts:915-919` (`to: txRequest.to`, `data: txRequest.data as \`0x${string}\``, `value: txRequest.value ? … : "0"`). The `minOut` shown in the receipt (`decoded.args.minOut`, L928) is only `formatUnits(quote.estimate.toAmountMin, …)` — a **display echo of LiFi's own number**, not a value the MCP computes or can re-derive locally. There is no MCP-side min-out arithmetic on the LiFi path to "fix" as the #685 body's option 1 assumes.
  - **Citation correction:** #722 cites `src/index.ts:918`. `src/index.ts:918` is inside the generic `handler()` wrapper (skill-pin plumbing), unrelated to swaps. The real verbatim-passthrough line is **`src/modules/swap/index.ts:918`** (`data: txRequest.data …`). Same line number, different file. The substance holds.
- **How the quote is requested today.** `prepareSwap` builds `lifiReq` (`src/modules/swap/index.ts:714-738`) and calls `fetchQuote` → `getQuote` (`src/modules/swap/lifi.ts:184-194`). The only slippage input is `slippage: args.slippageBps !== undefined ? args.slippageBps / 10_000 : undefined` (L723). When `undefined`, LiFi's SDK default (0.5%) applies. **No `fee` parameter is passed** anywhere — `createConfig({ integrator: "vaultpilot-mcp" })` (`lifi.ts:9-12`) sets the integrator string only. The 0.25% integrator fee is therefore configured server-side by LiFi for that integrator (or is a LiFi default); it is **not** set or seen by our code except as a line item in `quote.estimate.feeCosts` (already summed by `sumLifiCostsUsd`, `swap/index.ts:234-263`).
  - **Fee-source verdict (resolves the adversarial BLOCKER): OPAQUE — not an MCP-owned knowable constant.** Grounded against primary code: `createConfig({ integrator: "vaultpilot-mcp" })` (`swap/lifi.ts:9-13`) passes the integrator **label only** — no `fee` parameter and no `apiKey`; every `getQuote` call in `lifi.ts` passes `fromChain/toChain/fromToken/toToken/amount/fromAddress/slippage/filter-fields` and **no `fee`**; and a grep across `src/modules/swap/` + `src/config/` finds **no** LiFi integrator-fee constant anywhere (the only `apiKey`s in the tree are 1inch/etherscan/reservoir/tron/rpc — none is a LiFi credential). So the MCP calls LiFi unauthenticated and owns no fee rate: the 0.25% skim exists only in the returned `feeCosts`/calldata, and the code cannot re-derive it from a config. The KNOWN-CONSTANT branch (pad the quote by a config-known integrator rate, one round-trip) is therefore **ruled out** — there is no such constant to pad by, and none *should* be invented as a hardcoded literal (it would drift from LiFi's actual per-route fee). Consequence: the design takes the **opaque + fail-closed** path — it must never treat "no fee visible" as "no fee present" without a positive proof (Part 4). Forward note: if an operator ever configures a real LiFi integrator fee (via `createConfig`'s `fee` + a registered `apiKey`/dashboard), *that* rate becomes a single-home constant and the zero-extra-round-trip variant reopens — but that is not today's state.
- **The patch is irrelevant to the bug.** `patches/@lifi+sdk+3.16.3.patch` only stubs out the Sui re-exports (`Sui = undefined`, `isSui = () => false`) to keep `@mysten/sui` out of the bundle (~25 MB). It does **not** touch quote/fee/slippage logic. It does not affect #685.

**Verdict on the premise:** GROUNDED against primary code. The issue's "derive `_minAmountOut` from the net input" fix (option 1) is architecturally infeasible as stated — we hold no such value. The design must act at the **quote-request layer** (the one lever we own that shapes LiFi's baked min-out), which is exactly what #722 asks ARCH to decide.

## 2. Root cause

The revert mechanism, reconstructed from the #685 on-chain decode plus the LiFi Diamond ABI:

- The route (`swapTokensMultipleV3ERC20ToERC20`, selector `0x5fd9ae2e`) is a **GenericSwapFacetV3** entry point (`src/abis/lifi-diamond.ts:24,99`). Its ABI carries a **single top-level `_minAmountOut` (uint256)** after the four common inputs, plus a `_swapData[]` array where each leg has its own `fromAmount` (`lifi-diamond.ts:66-102`). GenericSwapFacetV3 measures the receiving-asset balance delta across all legs and reverts **`CumulativeSlippageTooHigh`** when the received amount `< _minAmountOut`.
- Leg 0 is a **FeeForwarder** (`forwardERC20Fees`, a token-unchanged pass-through `sendingAssetId == receivingAssetId == USDC`) that skims 0.25% (37.5 USDC) off the 15,000 USDC principal **before** the swap. Leg 1 (the real DEX swap successor) therefore receives only **14,962.5 USDC**. (Leg numbering is 0-based throughout this doc; the `requiresDeposit` flag is NOT the skim marker — §4.3.)
- The baked `_minAmountOut` decoded from the calldata is **14,985.92 USDT** — sized off the **gross 15,000** notional, not the post-fee 14,962.5. Even at a perfect 1:1 with zero DEX slippage, leg 2 can return at most ~14,962.5 USDT, i.e. **~23.4 USDT below the required min-out**. Unreachable by construction, independent of live price — matching the reporter's observation that the expected output did not move between attempts.
- **Grounding grade:** the *arithmetic* (14,985.92 baked min-out vs 14,962.5 net input) is DECODED from the failed tx's calldata — well-grounded. The error *name* `CumulativeSlippageTooHigh` is INFERRED from GenericSwapFacetV3 semantics (Etherscan shows only generic `execution reverted` because LiFi uses custom errors). The facet-name-to-error mapping is corroborated by the ABI's own source pin (`GenericSwapFacetV3.sol`). Residual: the exact effective-bps figure implied by 14,985.92 (~9 bps off gross) does not fully reconcile with LiFi's 50 bps default, so the precise LiFi-internal min-out formula is not fully pinned — see Part 4 R1/R2. This does not change the fix, which is robust to the internal formula (Part 4).

**One-line cause:** the on-chain min-out LiFi bakes is not loosened enough to absorb a principal skim (the integrator fee) that is deducted before the measured swap leg, so whenever `integratorFee > appliedSlippage` the swap reverts deterministically.

## 3. Options evaluated

Judged against: simplicity-first (fewest moving parts on the fund path), no latency regression vs R2 (`prepare_*` p95 ≤ 8 s, ASSUMED/UNMEASURED — ARCHITECTURE §2), and **no change to security posture** (the LiFi §3.4 B5-skip stays exactly as-is; any decode is for min-out correctness only, never a gate).

### Option A — fee-aware quote slippage (RECOMMENDED)

Compute an effective quote slippage that covers the principal-skimming fee and pass it to LiFi's quote request, so LiFi bakes a min-out reachable after the skim. The FIX lever stays the quote-request `slippage` we own — route-agnostic. The skim **magnitude** is now quantified by a **generic-swap calldata leg-walk** (per-pass-through-leg fractions summed off the decoded `_swapData[]`, §4.2), with structured `quote.estimate.feeCosts` as a **corroborating cross-check** (`feeFraction = max(decode, feeCosts)`). It is NOT sized from `feeCosts` alone: because the fee is opaque (Part 1) and `feeCosts` completeness is unverifiable (residual R2), a `feeCosts`-only threshold fails **open** — the adversarial BLOCKER. **The decode is load-bearing for quantification** — it drives how far to widen — but the fix MECHANISM is still the route-agnostic slippage lever; it applies to the **generic-swap class only** (the sole class with a same-denomination local reference). A bridge / cross-chain route is NOT gated by the reachability check: it **ships via its existing prepare path** (`verifyLifiBridgeIntent` / `assertCrossChainAddressing`, unchanged) and is INSTRUMENTED with a source-side-only suspected-unreachable counter (§4.2 step 6, PROD option (C)); it neither fails open (no cross-denominated ratio is ever computed) nor REFUSEs (no shipping cross-chain flow is regressed). An undecodable / unclassifiable generic-swap route fails **closed** (REFUSE at classification, §4.2 step 1c).

- Swap-class invariant (Part 5): **MEETS** — the returned min-out accounts for the fee skim while preserving the user's market-slippage tolerance exactly (effective = user + fee).
- Latency: **MEETS** — one *conditional* extra LiFi round-trip, only when a principal skim is present, plus one local ABI decode (cheap). Fee-free routes (decode-proven) take zero extra round-trips. Well inside R2's budget even in the two-round-trip case (one added `getQuote`). The zero-round-trip config-known-fee variant is ruled out (Part 1 — fee is opaque).
- Security posture: **UNCHANGED** — reads `feeCosts`, adds a read-only skim-detection decode reusing `lifiDiamondAbi` (the same ABI `decodeCalldata` already decodes on the signing path, `src/signing/decode-calldata.ts:545` — no new inspection-surface class), sets the `slippage` request param, and re-issues a read-only quote. The decode is min-out-correctness only, never a gate: no `classifyDestination`/B5 change, no new `allowedAbi:null` destination (§3.4 posture intact).
- Route-shape robustness: **MEETS with a scoped gate and an instrumented bridge path** — the `slippage`-widen FIX applies on the **generic-swap class only**, the sole class with a same-denomination local reachability reference (`estimate.toAmount`, the output token, same denomination as the baked `_minAmountOut`/`_minAmount`). A bridge (`tryDecodeLifiBridgeData` succeeds or `hasSourceSwaps` present) has **no** such reference (its `BridgeData.minAmount` is source-side; the destination reference is on another chain, unreadable at prepare time), so the reachability check does NOT run on it: the bridge **ships via its existing prepare path** (status quo) and is INSTRUMENTED with a source-side-only suspected-unreachable counter (PROD option (C), §4.2 step 6) — no cross-denominated ratio is ever computed (never fails open), and no shipping cross-chain flow is regressed (never REFUSEs). An unknown / new facet that is neither a recognized generic-swap selector nor a decodable bridge fails **closed** (REFUSE at classification, §4.2 step 1c). The 1inch-direct fallback is out of scope (computes its own min-out off net `dstAmount`, `swap/index.ts:635` — not subject to this bug).

### Option B — decode LiFi Diamond calldata + conditional re-quote (#722 scouting candidate)

Decode `transactionRequest.data` with `lifiDiamondAbi` for the baked `_minAmountOut` and the net DEX-leg `fromAmount`; re-quote with `slippage = feeFraction + userSlippage` only when baked `minOut > net-achievable`.

- Swap-class invariant: **MEETS on the covered route shapes only.**
- Latency: same conditional extra round-trip as A **plus** a local ABI decode (decode itself is cheap).
- Route-shape robustness: **FAILS.** `lifiDiamondAbi` enumerates only the seven generic-swap selectors. The decode does not cover: `swapAndStartBridgeTokensVia*` (source-swap-then-bridge — min-out lives in `BridgeData.minAmount` / facet-specific data, not the generic-swap `_minAmountOut`); any future/unknown facet selector; and the **1inch-direct fallback** (`prepareDirectOneInchSwap`, `swap/index.ts:581`) which has **no LiFi calldata at all**. On an unrecognized route the decode either fails-open (silently skips the fix → re-revert) or fails-closed (refuses a valid swap). Both are worse than A.
- Security posture: UNCHANGED if implemented as min-out-correctness-only — but it introduces a new calldata-inspection surface adjacent to the pre-sign gate, inviting future confusion with a security check (§3.4 warns against exactly this).

### Verdict

**Option A is recommended.** Named criterion: **the fix lever is the one we own route-agnostically — the quote-request `slippage`** — while the `vetLifiQuote` gate (§4) quantifies the widen on the one class it can prove reachable same-denomination (the generic-swap class), and every route it cannot prove that way is either shipped-and-instrumented (bridges — PROD option (C)) or REFUSED (an unclassifiable generic-swap / unknown facet). A widens `slippage` and re-quotes to correct the baked min-out on the **generic-swap class** (quantified by the leg-walk against `estimate.toAmount`, same denomination); it never fails open. Both A and B decode calldata, and in A the decode is **load-bearing for quantification** on the generic-swap class (it sizes the widen); the distinction from B is that A's fix MECHANISM is still the route-agnostic slippage lever scoped to the generic-swap class, whereas B makes the decode the SOLE fix (it computes the corrected min-out from calldata) with coverage bounded to the seven generic-swap selectors and **no** backstop — so on `swapAndStartBridge*`, unknown facets, and the 1inch-direct fallback (no LiFi calldata at all) B must either fail-open (re-revert) or over-refuse. Bridge / swap-and-bridge routes are **NOT** regressed under A + PROD option (C): they ship via their existing prepare path (status quo) and carry a source-side-only suspected-unreachable counter, and **#745** (build bridge reachability) is promoted on measured demand — one confirmed live bridge revert of this class or a >1% suspected-unreachable rate. Push back on B as a bounded-coverage decode made load-bearing for the fix with no backstop. Push back equally on any resurrection of a `BridgeData.minAmount` reachability ratio against a destination-side reference — it is cross-denominated and fails open (Amendment 3, §4.3); the option-(C) bridge signal stays strictly source-side (§4.2 step 6).

## 4. Recommended design (what DEV implements)

**A single uniform fail-closed gate — `vetLifiQuote` — is the SOLE producer of a shippable GENERIC-SWAP quote** (PROD option (C): the reachability gate is scoped to the same-chain generic-swap class only; it is NOT the sole producer for bridges — a bridge-classified route ships via its existing prepare path and is instrumented, §4.2 step 6). The gate replaces the earlier step-based sequence (old steps 2/2b/4), whose skim measurement drifted between the first-quote path and the re-quote-verify path: the classification and quantification ran on the first quote, but a thinner check ran on the re-quote, so a skim appearing only on the re-routed quote slipped through — the amendment-2 fail-open the adversarial REVIEW's BLOCKER exploited. The gate closes this **by construction**: the same classification catch-all and the same skim measurement run on EVERY gated generic-swap quote (q1 AND q2), and a re-quoted `swapTx` is built ONLY from the quote the gate returned.

### 4.1 The gate and its pipeline

```
vetLifiQuote(quote, sourceToken, userSlippage) → SHIP | REQUOTE(feeFraction) | REFUSE   // GENERIC-SWAP class only
```

Pipeline in `prepareSwap` (mirrored in `getSwapQuote` so the preview `toAmountMin` matches what `prepare_swap` bakes):

1. Fetch `q1 = await fetchQuote(lifiReq)` with the user's slippage as today.
2. **Classify `q1` by selector (§4.2 step 1), then branch by class:**
   - **Bridge class** (`tryDecodeLifiBridgeData` succeeds OR `hasSourceSwaps` present, and NOT a generic-swap selector) → the reachability gate does **NOT** run. Compute the **source-side-only** suspected-unreachable signal (§4.2 step 6), `console.warn` + increment the counter if it fires, and take **`q1`** into the existing prepare path unchanged (no re-quote). This is the status-quo bridge flow, now instrumented.
   - **Generic-swap class** → run `gate(q1)`:
     - `SHIP` → q1 is the final quote.
     - `REQUOTE(f)` → fetch `q2 = fetchQuote(lifiReq with slippage = userSlippage + f)`, then `gate(q2)`: `SHIP` → q2 is the final quote; anything else → **REFUSE**.
     - `REFUSE` → refuse with a clear error (do not return calldata that will deterministically revert).
   - **Neither** (unknown / new facet — not a recognized generic-swap selector and not a decodable bridge) → **REFUSE** (unclassifiable generic-swap, §4.2 step 1c).
3. **Exactly one re-quote** (generic-swap only). A `REQUOTE` verdict on `q2` never fires a third fetch; a still-unreachable `q2` REFUSES. The bridge branch never re-quotes.
4. Build `swapTx` from the class-appropriate quote — the **gate-returned** quote on the generic-swap class (never a discarded first quote), or **`q1`** on the bridge class — and place the **classification** (and, for generic-swap, the whole gate) **BEFORE** the existing guard block (`swap/index.ts:781–877`) so every current guard (`verifyLifiBridgeIntent`, `assertCrossChainAddressing`, decimals cross-check, exact-in `fromAmount`-drift refusal, >10× sanity) validates the FINAL quote. The bridge instrumentation is a read-only signal computed alongside — it NEVER preempts or replaces those existing bridge guards (unlike the deleted REFUSE-on-bridge, which sat before them and disabled every one).

The **classification catch-all (§4.2 step 1) running on q1 AND q2** is the load-bearing property for the generic-swap class: it is what closes the step-2-vs-step-4 drift the BLOCKER exploited. No generic-swap path ships a quote the gate did not classify and quantify; the bridge path ships the classified quote status-quo and records the source-side signal.

### 4.2 Gate internals

**The reachability primitive (denomination-safe — the ONE primitive).** A baked min-out is proven reachable ONLY by comparing it against a **same-token / same-decimals** locally-readable achievable reference `R`, via `appliedSlippage = 1 − bakedMinOut / R`. That ratio is dimensionless (a valid slippage) ONLY when `bakedMinOut` and `R` are the **same token**; across denominations it is arithmetic nonsense (a 6-dec source amount over an 18-dec destination amount reads as a ~10¹²-scale "slippage"). The classification below is exactly the decision of **whether such an `R` exists**:

- **Generic-swap class** — `R = estimate.toAmount`, the OUTPUT token, the same denomination as the baked `_minAmountOut` (six V3 selectors) / `_minAmount` (legacy `swapTokensGeneric`). The primitive is well-defined → run the sound reachability check (steps 2–5). This class is UNCHANGED from the sound prior design.
- **Bridge / swap-and-bridge class** — **no same-denomination local `R` exists**. A bridge's baked min-out is `BridgeData.minAmount`, denominated in the SOURCE asset (`sendingAssetId`, `lifi-diamond.ts:122–123`), while any achievable reference for the amount the user *receives* is on the DESTINATION chain, unreadable at prepare time. With no valid `R` the primitive cannot be evaluated, so the reachability check does **NOT** run — the route ships via its existing prepare path (status quo) and is instrumented with a source-side-only suspected-unreachable signal (step 6). **No cross-denominated ratio is ever computed.** (PROD option (C): the bridge is not REFUSED — REFUSE-on-bridge would disable every shipping EVM cross-chain flow.)
- **Unknown / new facet** (neither a recognized generic-swap selector nor a decodable bridge) — also no valid `R`, and no existing bridge path to fall through to, so it **REFUSEs** (step 1c) rather than ship an unproven min-out. **No cross-denominated ratio is ever computed.**

**Invariant — no unstated complement (the feeCosts-only fail-open cannot re-emerge).** Every path the **gate** ships (i.e. every generic-swap SHIP) runs this same-denomination primitive to an `appliedSlippage ≥ feeFraction ≥ 0` proof. **NO branch may derive a reachability proof from `feeCosts` alone and ship on it** — `feeCosts` is only ever a corroborating `max()` against the decoded leg-walk (step 3), never a standalone reachability signal, and it exists only inside the generic-swap class. State this so the next new route shape cannot reopen the hole: **the gate's reachability-SHIP is generic-swap only** — no other class is ever SHIPPED on a `feeCosts`-derived (or any cross-denominated) reachability proof. A bridge ships via its EXISTING prepare path (status quo, as on `main`), NOT through the gate and NOT on any reachability proof — it makes no reachability claim at all, and its source-side `feeCosts` reading feeds only the read-only suspected-unreachable counter (step 6), never a ship-or-refuse decision.

1. **Classify by SELECTOR, never by `to`.** Pinning `to == LIFI_DIAMOND` would breach §3.4 (no new trusted destination), so classification reads the calldata selector:
   - (a) one of the **seven generic-swap selectors** (`lifi-diamond.ts:94–102`) → **leg-walk** (steps 2–5), the same-denomination class.
   - (b) **NOT a generic-swap selector AND `tryDecodeLifiBridgeData` succeeds** (`decode-calldata.ts:203–231`), or the decoded `hasSourceSwaps` flag is present → **bridge class → existing prepare path + source-side instrumentation** (step 6), NOT REFUSE. Selector-first ordering is load-bearing: a generic-swap calldata is classified at (a) and never reaches the positional bridge decode, which would otherwise mis-read it (viem decodes the leading tuple and ignores trailing bytes).
   - (c) neither → **REFUSE**. An unknown / new LiFi facet selector lands here by construction. Its REFUSE copy, and the generic-swap topology REFUSE at step 2, obey the **same-chain REFUSE copy rule (PROD condition 4): do NOT steer the user to `prepare_custom_call`** — see step 5.
   This catch-all runs on EVERY gated (generic-swap) quote (q1 AND q2) — it is what kills the amendment-2 drift hole; the same-denomination primitive above is what kills the cross-denomination hole.
2. **Leg-walk (generic-swap class).** Normalize `_swapData` to an array (single-* selectors carry one tuple, multi-* an array). Require topology, else **REFUSE** (splits / gaps / incoherent route the gate cannot reason about):
   - `leg[0].sendingAssetId == sourceToken` (native → the zero/native sentinel LiFi encodes in `sendingAssetId`; DEV confirms the exact sentinel),
   - `leg[0].fromAmount == action.fromAmount`,
   - `leg[i+1].sendingAssetId == leg[i].receivingAssetId` for all `i`.
   **Fee-leg marker = `sendingAssetId == receivingAssetId`** — a *pass-through* leg whose token is unchanged can only forward or skim (a DEX / wrap leg always changes the token). Per pass-through leg that has a successor: `f_i = 1 − next.fromAmount / leg_i.fromAmount` (bigint, **ROUNDED UP**); `f_i < 0` → **REFUSE**; a pass-through leg as the LAST leg → **REFUSE** (its skim is unsizeable — no successor to measure against). `feeFraction_decode = Σ f_i` — a provable over-estimate, correctly denominated because each `f_i` is a same-token ratio.
3. **`feeCosts` corroboration (never a standalone ship signal — invariant above).** Sum `feeCosts[]` entries whose `token.address` equals the source token (native when `fromToken === "native"`), over `BigInt(action.fromAmount)`; `feeFraction = max(feeFraction_decode, feeFraction_feeCosts)`, **ROUNDED UP**. Fees denominated in the output token or gas do not skim principal and are excluded. This requires widening the local `LifiCostLike` (`swap/index.ts:228–232`) to carry `token.address` — its absence today is the R2 trigger, and until it is widened this signal cannot even run. **Consult each entry's `included` flag:** a fee with `included: false` is charged on top of, not skimmed from, the principal, so it does not reduce the measured swap input and excluding it is correct. Getting `included` wrong in the safe direction — counting an `included: false` fee anyway — **over-widens only (never fails open)**, so a DEV who cannot yet read `included` may conservatively count all source-token entries; but consult it to avoid needless MEV give-up.
4. **Baked min-out and applied slippage (generic-swap class only).** The baked min-out is the **DECODED** generic-swap min-out field — `_minAmountOut` on the six V3 selectors, `_minAmount` on the legacy `swapTokensGeneric` — **never** `estimate.toAmountMin` until R3 confirms they are equal (the chain reverts against the calldata-baked value, not the structured field). `appliedSlippage = 1 − minOutBaked / estimate.toAmount`, **ROUNDED DOWN** — a valid dimensionless slippage because `minOutBaked` and `estimate.toAmount` are the **same** (output) token. The **opposed rounding** — `feeFraction` UP, `appliedSlippage` DOWN — makes both err toward REFUSE. This is spec, not accident: float/bigint drift near equality is a classic false-pass. **Bridge routes never reach this step — they took the existing prepare path at step 1b (not gated)**, so `BridgeData.minAmount` is never fed into this ratio (that would be cross-denominated — Amendment 3, §4.3).
5. **Decision (generic-swap class).**
   - No pass-through legs **AND** `feeFraction == 0` **AND** `appliedSlippage ≥ 0` (i.e. `minOutBaked ≤ estimate.toAmount`) → **SHIP** (proven clean, zero extra round-trips). The `appliedSlippage ≥ 0` clause is not redundant: a malformed quote with `minOutBaked > estimate.toAmount` (negative `appliedSlippage`) is already unreachable at the quote rate even with **no** skim detected, and MUST **REFUSE**, not ship. Fee-absence alone is never sufficient — reachability is.
   - Else `appliedSlippage ≥ feeFraction` → **SHIP**. **Reachability, not fee-absence, is the ship criterion**: a first quote whose LiFi-baked min-out already absorbs the skim ships without a re-quote, and this also caps double-widening / MEV give-up when LiFi's `toAmount` is already net-of-fee. (Since `feeFraction ≥ 0`, this branch already implies `appliedSlippage ≥ 0`.)
   - Else → **REQUOTE(feeFraction)** if this is q1, **REFUSE** if this is q2.
   - **REFUSE copy (PROD condition 4) — does NOT steer to `prepare_custom_call`.** Any generic-swap REFUSE here — a `q2` still unreachable, a negative `appliedSlippage`, an incoherent-topology route (step 2), or an unclassifiable / unknown facet (step 1c) — states that the route's baked min-out cannot be proven reachable at prepare time and is refused, and MUST NOT suggest `prepare_custom_call` as a workaround: `prepare_custom_call` bypasses the pre-sign selector check (block 5) behind a single ack **without** solving reachability — the same `CumulativeSlippageTooHigh` revert still fires, now with this gate silenced and the safety check skipped. The copy says so explicitly. Route the user to re-run `get_swap_quote` (fresh route) instead.
6. **Bridge class → SHIP via existing path + source-side instrumentation (PROD option (C)).** A route that classifies as a bridge at step 1b — `tryDecodeLifiBridgeData` succeeds or `hasSourceSwaps` is present, covering BOTH pure bridges (`startBridgeTokensVia*`) and `swapAndStartBridgeTokensVia*` source-swap-then-bridge routes — **ships via its existing prepare path unchanged** (`verifyLifiBridgeIntent` / `assertCrossChainAddressing` / decimals cross-check / intermediate-chain machinery), and additionally emits a **source-side-only** suspected-unreachable signal. The reachability gate does NOT run on it: there is no `BridgeData.minAmount` reachability check, no opportunistic `[BridgeData, SwapData[]]` decode against a destination reference, and no leg-walk against a bridge — **all cross-denominated dest-side math stays DELETED** (§4.3). PROD option (C): REFUSE-on-bridge is downgraded to a counter, because refusing here would disable every shipping EVM cross-chain flow to guard a bridge revert that has not been observed (the #685 report is a same-chain generic-swap fee case).
   - **Source-side-only suspected-unreachable signal (PROD condition 1 — the FRAMEWORK; the exact predicate is a DEV live-verify item, R6).** Compute the signal from **denomination-safe SOURCE-SIDE data ONLY** — every operand denominated in the SOURCE token: `BridgeData.minAmount` (source-side, sits with `sendingAssetId`), `action.fromAmount` (source-side), and source-token `feeCosts[]` entries (`token.address == sourceToken`). **Flag** when a detectable source-side skim is present that the source-side amounts do not appear to account for — e.g. a source-token `feeCosts` skim (or a `hasSourceSwaps` source-leg amount discontinuity) whose magnitude the baked `BridgeData.minAmount` does not appear to absorb relative to `action.fromAmount`. **Do NOT compute the invalid cross-denominated dest-side ratio** (`1 − minAmount/estimate.toAmount`); **do NOT REFUSE**. The design fixes the FRAMEWORK — source-side-only signal + counter + never-REFUSE + never-cross-denominated; **DEV validates the precise predicate against live fee-bearing bridge quotes** (R6). Because every operand is source-token-vs-source-token the comparison is dimensionally valid; because it never touches the destination amount it cannot fail open on a decimals-up bridge.
   - **LOG + COUNT (PROD condition 1 — "gate downgraded to a counter").** On a flagged route, emit a structured stderr line — `console.warn("[vaultpilot-mcp] lifi.bridge_suspected_unreachable", { fromChain, toChain, bridge, hasSourceSwaps, destinationChainId })` (stderr, not stdout — stdout is the MCP stdio channel; this matches the existing `console.warn`/`console.error` diagnostics in `src/index.ts`) — and increment a process-lifetime monotonic counter `lifiBridgeSuspectedUnreachableCount` exported from the swap module. This is a read-only diagnostic: the route still SHIPS. DEV binds the counter to whatever metrics surface exists at implementation time; until one does, the stderr line plus the module counter is the minimum countable signal, and the fixture-replay tally (§6) reads it.
   - **#745 promotion trigger (PROD condition 2).** **ONE confirmed live bridge revert of this class OR a >1% suspected-unreachable rate** (flagged bridges / total bridge prepares) → promote **#745** (build a destination-chain-aware bridge reachability mechanism) and **reopen the disable question** (whether to REFUSE bridges after all). Until either fires, bridges ship status quo — the counter is the evidence #745 is built on, not a pre-emptive disable.
   - **No blast radius on the existing bridge guards (the reason (C) beats REFUSE-on-bridge).** Because the bridge route takes the existing prepare path, `verifyLifiBridgeIntent`, `assertCrossChainAddressing`, and the intermediate-chain-bridge (NEAR-Intents) machinery all still run **unchanged** — the EXISTING `prepare_swap` cross-chain flows (EVM→EVM bridges, **EVM→Solana**, **EVM→TRON**, intermediate-chain routes) are **NOT regressed**. That is exactly the blast radius REFUSE-on-bridge would have caused and (C) avoids: the deleted rule sat BEFORE the guard block and made every one of those flows REFUSE. DEV MUST NOT place the instrumentation before those guards, and MUST NOT let it throw — it is a counter, not a gate. The `src/modules/{tron,solana,btc}/lifi-swap.ts` sibling modules remain a SEPARATE surface (their own follow-up — see Cross-chain siblings), unaffected by this EVM-path change.
   - **Preview (`getSwapQuote`) — SOFT annotation, not a throw (PROD condition 5).** `getSwapQuote` is read-only and produces no signable calldata, so a suspected-unreachable bridge route returns a **non-fatal annotation** ("min-out may be unreachable on this cross-chain route; bridge reachability is pending #745") — never an exception. A hard throw on a read-only preview would degrade UX for no safety gain (nothing is signed). `getSwapQuote` mirrors the classification and the source-side signal, and returns LiFi's `toAmountMin` as-is for a bridge (there is no reachability adjustment to make).
7. **Unchanged invariants.** Overflow (`userSlippage + feeFraction` over LiFi's max accepted slippage) → **REFUSE**, never clamp below `feeFraction` (that re-reverts). The ack gate — `assertSlippageOk` (`swap/index.ts:357`) plus the >100 bps sandwich-ack — runs on the **USER's** market slippage only, NOT `effectiveSlippage`; the fee pad is a deterministic skim, not sandwich exposure, and must not trip the ack. The receipt discloses `effectiveSlippageBps` / `integratorFeeBps` (display-only; changes no signed bytes beyond the corrected min-out). **Exact-out needs NO carve-out** — the gate never branches on `amountSide`, so it subsumes the old exact-out residual. If the re-quote comes back *cleaner* (`feeFraction₂ < feeFraction₁`), ship and disclose — the over-width is bounded by `feeFraction₁` and the iteration is single.

### 4.3 What is REMOVED from the earlier design, and why it was wrong

- **The "real DEX leg" wording is REMOVED.** "The real DEX leg" is undefined for a multi-leg pipeline, and picking the wrong leg under-counts: a source-token leg reads `skim = 0`, and an intermediate-token leg makes the ratio cross-denominated nonsense. Both quotes would run the same broken measurement and self-consistently false-pass. The leg-chain-walk with per-pass-through-leg fractions summed (§4.2 step 2) replaces it.
- **The `requiresDeposit` fee marker is REMOVED.** `requiresDeposit` is **not** a fee marker: the first leg (leg 0) of EVERY route sets `requiresDeposit:true` (the facet pulls the user's tokens on deposit), so a benign deposit-then-DEX route with no fee sets it too — the flag fires on every legitimate route (including #685's own FeeForwarder, which happens to be that leg-0 deposit leg) and does not structurally isolate a skim. The structural marker is `sendingAssetId == receivingAssetId` (token-unchanged pass-through): a leg that forwards its own token can only pass through or skim, whereas a DEX / wrap leg always changes the token.
- **The cross-denominated bridge math is REMOVED (Amendment 3 — the structural fix; STANDS under Amendment 4).** The opportunistic `[BridgeData, SwapData[]]` positional decode against a destination reference, the cross-denominated `BridgeData.minAmount / estimate.toAmount` reachability check, and the `hasSourceSwaps === false → ship` branch are all DELETED. `BridgeData.minAmount` is denominated in the SOURCE asset (`sendingAssetId`, `lifi-diamond.ts:122–123`) while `estimate.toAmount` is the DESTINATION output — the ratio `1 − minAmount/estimate.toAmount` is dimensionally invalid and, on a decimals-up bridge (source 6-dec → destination 18-dec), yields a spuriously huge `appliedSlippage` that **fails OPEN** (ships an unreachable min-out that reverts). There is no local same-denomination DESTINATION reference for a bridge, so the class computes **NO reachability ratio at all**: under PROD option (C) it ships status quo and carries a **source-side-only** suspected-unreachable counter (§4.2 step 6) — the source-side signal (`BridgeData.minAmount` vs `action.fromAmount` vs source-token `feeCosts`, all source-denominated) is dimensionally valid and never touches the destination amount. Deleting the cross-denominated formula is the STRUCTURAL fix; whether a bridge then REFUSEs (Amendment 3) or ships-and-instruments (Amendment 4) is the separate PROD scope decision — (C) chose the latter to avoid regressing every shipping cross-chain flow.

### 4.4 §3.4 posture and where it lives

§3.4 posture is **UNCHANGED**: the gate reuses `lifiDiamondAbi` and `tryDecodeLifiBridgeData` (the ABI + parse the signing path already runs, `src/signing/decode-calldata.ts:545`), pins no destination, and touches no `classifyDestination` / B5 / `allowedAbi`. It is min-out-correctness only, never a gate on the pre-sign path.

**Where it lives:** `vetLifiQuote` and its leg-walk helper live on the LiFi quote-request path in `prepareSwap` / `getSwapQuote` (`src/modules/swap/index.ts`) — NOT the shared min-out helper (Part 7). Factor the gate as one function shared by `prepareSwap` and `getSwapQuote` so preview and prepare stay consistent. Widen the local `LifiCostLike` interface (`swap/index.ts:228–232`) with `token.address` first — signal (a) cannot run without it.

## 5. Swap-class invariant it must satisfy

**LiFi-baked min-out reachability, provable ONLY same-denomination.** For any `prepare_swap` on the **generic-swap class** — a route whose baked min-out (`_minAmountOut`/`_minAmount`) is denominated in the output token, the SAME token as `estimate.toAmount` — the prepared transaction's on-chain min-out MUST be reachable at the quote-time rate after every principal skim taken before the measured swap leg (integrator fee, any pre-swap DEX/wrap fee) — i.e. the effective slippage requested of LiFi satisfies `effectiveSlippage ≥ Σ(principalSkimFraction) ≥ 0` and the resulting baked min-out cannot deterministically revert on slippage — **while not over-loosening**: the market-slippage give-up beyond the skims stays bounded by the user's requested tolerance (`effectiveSlippage − Σ(skims) = userSlippage`). The tx must neither deterministically revert (min-out too high) nor silently widen MEV exposure beyond `userSlippage + fees-actually-charged` (min-out too low). For a **bridge / swap-and-bridge class** there is **no same-denomination local reference** to prove reachability against — the baked `BridgeData.minAmount` is source-side and the achievable reference is on the destination chain — so the gate proves **no** reachability for it: under PROD option (C) the bridge ships via its existing prepare path (status quo, the flows already in production) and carries a **source-side-only** suspected-unreachable counter (`BridgeData.minAmount` / `action.fromAmount` / source-token `feeCosts`, all source-denominated), and **#745** is promoted on measured demand. An **unknown / new facet** with no valid reference and no existing bridge path to fall through to **REFUSEs**. A min-out **MUST NEVER be shipped on a cross-denominated ratio**, and no bridge signal may read the destination amount.

Falsifier for the invariant: the acceptance test in Part 6.

## Part 4 — Adversarial self-check (residual risks)

Two fail-opens are now closed **by construction**. The drift-class fail-open — the same classification catch-all and skim measurement run on q1 AND q2 of the generic-swap class, and a re-quoted `swapTx` is built only from the gate-returned quote. The **cross-denomination fail-open** — a bridge computes **no** reachability ratio (§4.2 step 6): it ships via its existing prepare path (status quo) and carries only a source-side-only suspected-unreachable counter, so no cross-denominated `minAmount/toAmount` ratio is ever computed, and every generic-swap SHIP runs the same-denomination primitive (§4.2). What remains are proof-limits requiring DEV live-verification, not structural holes. Every generic-swap proof-limit fails toward REFUSE; the bridge path never computes a cross-denominated ratio and never ships a min-out derived from one — its only prepare-time surface change is a read-only counter.

**R1 — does LiFi's `slippage` param monotonically lower the baked on-chain min-out? (DEV live-verify; load-bearing for the fix.)** Corroborated by the #685 body's own workaround ("widen `slippageBps` enough to absorb the fee") and by the code's existing `slippage → toAmountMin` assumption, but NOT independently verified against the live API. **Safe under the opposite assumption:** the gate re-runs on q2 and asserts `appliedSlippage ≥ feeFraction` against the DECODED baked min-out — if the widened slippage did not lower it, the gate REFUSES rather than shipping a revert. Keep as a live check.

**R2 — RESOLVED into the gate (was the adversarial BLOCKER).** The original design derived the fail-closed threshold **solely** from `estimate.feeCosts`, whose completeness R2 doubts; an omitted source-token skim gave `feeFraction=0` → fail-OPEN ship of the #685 revert. Grounded trigger: `LifiCostLike` (`swap/index.ts:228-232`) has **no `token.address`**, so source-token matching cannot even run until DEV widens the type. **Resolution:** the leg-walk decode is now the RELIABLE skim source and `feeCosts` is demoted to a cross-check (`feeFraction = max(decode, feeCosts)`, §4.2 step 3); a route whose skim cannot be quantified REFUSES. DEV still confirms the real `@lifi/types` `FeeCost` fields against the installed package, but correctness no longer DEPENDS on `feeCosts` completeness.

**R3 — is `estimate.toAmountMin` the value that drives the on-chain revert? (DEV live-verify; blocking for the generic-swap class.)** The chain reverts against the calldata-baked generic-swap min-out (`_minAmountOut`/`_minAmount`), not the structured `toAmountMin` (bridges are not gated on reachability, so no bridge min-out is ever read into the ratio). The gate reads the **DECODED** baked value (§4.2 step 4), so it is correct even if the two diverge; R3 is the check that would let a later optimization trust `toAmountMin`. On a real fee-bearing quote, decode `transactionRequest.data` and assert the decoded baked min-out equals `estimate.toAmountMin`. Until confirmed, the gate stays on the decoded value.

**R4 — RETIRED (Amendment 3; stays retired under Amendment 4).** The former `hasSourceSwaps === false → SHIP` bridge residual is gone because there is **no bridge reachability claim at all now**. Under PROD option (C) a bridge ships via its existing prepare path exactly as on `main` (no reachability proof, as today) and the only prepare-time addition is a read-only source-side counter — no `BridgeData.minAmount` arithmetic ever gates a ship decision, so the residual it described (shipping on a `BridgeData.minAmount` reachability statement that rests on contract behavior) cannot occur. The bridge reachability that R4's SHIP branch tried to assert is deferred wholesale to **#745**; today the bridge path makes no such claim.

**R5 (restated tighter) — the linchpin is "every principal skim is visible as an amount discontinuity in the encoded `_swapData[]` OR a source-token `feeCosts` entry."** A skim hidden inside a token-CHANGING executor step, with downstream `fromAmount`s encoded as if no skim occurred, is invisible in calldata by construction — no local check closes this, for anyone. #685's own decode (leg 2 encoded at exactly 14,962.5 = post-skim) shows LiFi DOES encode skims visibly. **DEV falsifier:** on real fee-bearing quotes, assert the decoded skim ≥ the `feeCosts` skim and that the arithmetic reconciles with the baked min-out.

**R6 — the bridge suspected-unreachable predicate is a DEV live-verify item (PROD option (C), §4.2 step 6).** The DESIGN fixes the FRAMEWORK — the bridge signal reads **source-side data only** (`BridgeData.minAmount` / `action.fromAmount` / source-token `feeCosts`, all source-denominated), NEVER computes a cross-denominated dest-side ratio, NEVER REFUSEs, and only logs + counts. The EXACT predicate (what magnitude of un-absorbed source-side skim flags "suspected unreachable", and the threshold) is NOT pinned at design time — DEV validates it against live fee-bearing bridge quotes so the counter tracks real reverts, not noise. Fails safe by construction: a wrong predicate only mis-counts a read-only diagnostic — it can never REFUSE a shipping bridge or ship an unreachable min-out (the route ships status quo regardless of the signal). Escalation is measured, not predicted: **one confirmed live bridge revert of this class OR a >1% suspected-unreachable rate promotes #745** and reopens the disable question (§4.2 step 6).

**Exact-out — subsumed, no carve-out.** The gate never branches on `amountSide`, so exact-out routes flow through the same classification + leg-walk + reachability check. The old exact-out residual (scope the fix to exact-in, flag exact-out separately) is removed. The approval max-in cover (`applyMaxIn`, `swap/index.ts:967`) is unchanged and orthogonal.

**1inch-direct — verified OUT OF SCOPE.** `prepareDirectOneInchSwap` derives its own min-out from the net quoted `dstAmount`; `/swap` sets only `src`/`dst`/`amount`/`from`/`slippage` (`oneinch.ts:95-103`) with no fee/referrer parameter, so it is not subject to the provider-baked-off-gross bug and never reaches the gate.

**Opportunistic dest-side bridge reachability decode — DELETED (Amendment 3; stays deleted under Amendment 4).** The `[BridgeData, SwapData[]]` positional decode that fed a cross-denominated `BridgeData.minAmount` (source-side) / `estimate.toAmount` (destination-side) reachability ratio is removed entirely — it fails open on any decimals-up bridge, and no `lifinance/contracts` SwapData-layout dependency remains. Note the distinction under option (C): the existing `tryDecodeLifiBridgeData` still runs (it always did — classification + the source-side signal reads `BridgeData.minAmount` from it), but ONLY against source-denominated operands (`action.fromAmount`, source-token `feeCosts`); no destination-side value is ever read. Bridges ship status quo + a source-side-only counter (§4.2 step 6); there is no dest-side reachability decode to get right or wrong.

**Preserved design constraints (now gate steps, not residuals).** Hidden write paths: none — the gate reads `feeCosts`, decodes calldata read-only (the same parse `decodeCalldata` runs on the signing path, `src/signing/decode-calldata.ts:545`), sets a request param, and re-issues a read-only `getQuote`; the only signed-bytes change is the corrected min-out. Ack-gate split (§4.2 step 7) and guard ordering (§4.1 step 4) are folded into the gate — reviewer confirms both.

**Residual-risk summary.** Two fail-opens are closed by construction: the drift-class fail-open (uniform gate, catch-all on q1 AND q2 of the generic-swap class) and the cross-denomination fail-open (a bridge computes no reachability ratio at all — it ships status quo with only a source-side-only counter; every generic-swap SHIP runs the same-denomination primitive). R1/R3/R5 are DEV live-verify mandates for the generic-swap class, each failing toward REFUSE (R4 is retired); R6 is the DEV live-verify mandate for the bridge source-side proxy, failing toward mis-counting a read-only diagnostic — never a ship-or-refuse. The things DEV MUST NOT drop: (a) the classification catch-all on BOTH gated generic-swap quotes; (b) the `sendingAssetId == receivingAssetId` fee-leg marker (NOT `requiresDeposit`); (c) the opposed rounding (feeFraction UP, appliedSlippage DOWN) and the DECODED baked min-out; (d) the denomination-safe primitive — every generic-swap SHIP compares a same-token baked min-out against `estimate.toAmount`, and **NO `feeCosts`-only ship path**; (e) the bridge-class SHIP-via-existing-path with its **source-side-only** `lifi.bridge_suspected_unreachable` / `lifiBridgeSuspectedUnreachableCount` signal — **NO REFUSE**, and **NO resurrected cross-denominated `BridgeData.minAmount`/`estimate.toAmount` math** (the instrumentation must NOT sit before, or throw within, the existing bridge guards); (f) the ack-gate-on-user-slippage split; (g) the R1/R3/R5/R6 live checks; (h) the same-chain REFUSE copy that does NOT steer to `prepare_custom_call`.

## Over-refusal (bounded) and ops note

- **Same-chain generic-swap (the dominant #685 class):** near-zero refusal — the class the fix fully covers.
- **Cross-chain bridges and swap-and-bridge routes:** **status-quo — SHIP, instrumented, #745 on measured demand** (§4.2 step 6, PROD option (C)). They are NOT regressed: a bridge takes its existing prepare path (`verifyLifiBridgeIntent` / `assertCrossChainAddressing` unchanged) and carries a **source-side-only** suspected-unreachable counter. The prior opportunistic `[BridgeData, SwapData[]]` dest-side reachability decode stays DELETED (it rested on the cross-denominated `minAmount/estimate.toAmount` ratio that fails open on decimals-up bridges), so no bridge min-out is ever proven or refused at prepare time — the source-side counter is a diagnostic, not a gate. A rising `lifiBridgeSuspectedUnreachableCount` (or one confirmed live bridge revert of this class) is measured demand and the trigger to promote **#745** (a destination-chain-aware reachability mechanism) and reopen the disable question. Rationale: REFUSE-on-bridge would have disabled every shipping EVM cross-chain flow (EVM→EVM, EVM→Solana, EVM→TRON, intermediate-chain) to guard an unobserved bridge revert — (C) fixes the reported same-chain bug without that regression and measures the bridge risk instead of pre-empting it.
- **Ops note (standing risk).** If LiFi ships a new `GenericSwapFacetV4` / new generic-swap selectors, an unrecognized selector is neither a known generic-swap selector nor a decodable bridge, so it lands in the classifier catch-all (§4.2 step 1c) and REFUSES until `lifiDiamondAbi` (`src/abis/lifi-diamond.ts`) is extended. This is the acceptable REFUSE-worst-case, but the PR MUST carry this ops note so the failure is recognized fast — a sudden same-chain REFUSE spike means a missing selector, not an attack. (A `lifiBridgeSuspectedUnreachableCount` spike, by contrast, is not a refusal at all — bridges keep shipping; it is the measured-demand signal for #745.)

## Cross-chain siblings — out of scope, filed separately

`src/modules/{tron,solana,btc}/lifi-swap.ts` all forward LiFi `transactionRequest` artifacts **VERBATIM** under the same `vaultpilot-mcp` integrator with NO reachability proof — the SAME #685 bug class. Solana's `VersionedTransaction` and BTC's PSBT cannot run this EVM leg-walk gate at all, so a separate mechanism is needed there; whether the skim actually fires on those paths is unverifiable without live quotes. Tracked as a follow-up issue filed alongside this PR — **NOT blocking #685** (tracking issue filed alongside this PR).

## Evidence that would change the verdict

1. A live generic-swap quote whose downstream `fromAmount`s do NOT reflect a fee that `feeCosts` also omits → breaks R5 → false-pass.
2. A common EVM pair that LiFi routes as a bridge-facet call the user expected to be a same-chain swap → **ships today** (status quo, no same-denomination reference to prove reachability, so no proof attempted), and surfaces #745 demand via `lifiBridgeSuspectedUnreachableCount` if a source-side skim is detected — a product signal, not a REFUSE and not a fail-open (no cross-denominated ratio is computed).
3. A generic-swap selector LiFi adds without a corresponding `lifiDiamondAbi` entry → the route REFUSES until the selector is added (Ops note); a same-chain REFUSE spike is the tell.
4. R1 false → the gate still refuses correctly, but the FIX stops fixing (remedy is outside this design).
5. LiFi returning non-Diamond, non-decodable calldata on common EVM routes → REFUSE at classification; if frequent, a product decision.
