# Design: #759 ‚Äî recipient/authority authorization seam at pre-sign

Status: ARCH-CONVERGED DESIGN ARTIFACT, v4 ‚Äî awaits a fresh SEC + REVIEW verdict on this revision before DEV implements. Round 3 (authoritative) returned REVIEW: REQUEST_CHANGES (6 grounding-edge defects) and SEC: CHANGES_REQUIRED (blockers B1-B5, required conditions C1-C6, plus 14 doc-defects) against the prior revision (PR #765, evidence base post-#758 `main` `3d5ca3c`). **Both verdicts affirm the design's shape is sound** ("strongest security work in the repo's record" ‚Äî SEC) ‚Äî every finding is a correction inside the shape, not a redesign. v4 below is that revision with every accepted finding folded in.
Scope: fund-path (EVM pre-sign gate). Closes the #757/#760 incident class.
Evidence base: recon @ `origin/main` `32e001b`, cross-checked through three adversarial review rounds against `main`, most recently `3d5ca3c`. `src/signing/pre-sign-check.ts` is unchanged across that range (still 404 lines; every cited line below was re-checked live and matches). `src/modules/execution/index.ts` is **not** stable ‚Äî the ARCHITECTURE.md ¬ß5.2 decomposition (#720, PR #758) landed after this design's original evidence base and shrank it from 4013 to 2805 lines, moving `runEvmPreSignGuards`/`previewSend`/`sendTransaction` to new line numbers. Every `execution/index.ts` citation below is an **evidence pointer against the cited commit, not a normative anchor** (same convention as ARCHITECTURE.md ¬ß3's own header) ‚Äî DEV re-resolves each named symbol against current `HEAD` before implementing; the symbol names, not the line numbers, are load-bearing.

## 1. Problem & threat model

**Threat actor.** A rogue or prompt-injected agent ‚Äî in-model for this repo, which treats all agent-supplied input (tool arguments, `prepare_*` call shape, even which tool is called) as attacker-influenced. No user social-engineering is assumed beyond the user's ordinary Ledger confirm; the device screen is the only thing the threat model treats as trustworthy, and only for what it actually renders.

**The structural gap.** `assertTransactionSafe`'s block 5 (`src/signing/pre-sign-check.ts:338-403`) validates **which function is being called** ‚Äî the 4-byte selector must belong to the destination's curated ABI ‚Äî and never validates **where the value or authority goes**. For any recognized destination (`classifyDestination` returns non-null), block 4's argument-agnostic catch-all (`if (!dest)`, `pre-sign-check.ts:336`) is skipped entirely, because it only fires on *unrecognized* destinations. Block 5 then passes any selector on the ABI with no `decodeFunctionData`, no argument read, on the recognized-destination path. The result inverts the intended security model: being on the protocol allowlist makes a recipient-bearing call **more** likely to reach signature than an unrecognized one, not less ‚Äî an unknown destination is refused unless acked; a recognized one with a free `to`/`receiver`/`owner` argument is not gated at all.

**Asset at risk.** Whatever the recognized protocol call moves: Aave collateral (`withdraw.to`), Morpho supplied/borrowed assets (`withdraw`/`borrow`/`withdrawCollateral.receiver`), Uniswap LP/swap output (`mint`/`collect`/`exactInput*`/`exactOutput*.recipient`), Lido withdrawal NFTs (`requestWithdrawals.owner`), Permit2-authorized third-party token balances (`transferFrom.to`), ERC-4626 vault shares/assets (`deposit`/`mint`/`withdraw`/`redeem.receiver`), and self-funded credit positions (Aave `supply.onBehalfOf`, Morpho `supply`/`supplyCollateral.onBehalf`). No acknowledgement flag anywhere in the chain is load-bearing against this ‚Äî the schema-required `acknowledgeNonProtocolTarget: true` literal cannot distinguish a benign call from a drain because every call sets it, and per block 4's skip, the pre-sign layer does not even consult it on this path.

## 2. The failure class ‚Äî one substrate, four incidents

This is not a new bug class; it is the fourth occurrence of the same substrate defect surfacing in a different corner of the recognized-destination surface:

- **#711** (closed ‚Üí PR #727) ‚Äî `applyCustomCallClassifier`'s `transferFrom` ack-bypass checked only `args[0]` (the `from`); the recipient `args[1]` (`to`) was never inspected, so an ack-stamped `transferFrom(from=X‚âÝwallet, to=ATTACKER)` rode through unchecked.
- **#737** (sibling, found at #727's review) ‚Äî the *same* classifier's `transfer(address to, uint256)` path, which moves the **wallet's own** tokens, could still be ack-bypassed to an arbitrary recipient.
- **#741** (untracked sibling, found by REVIEW's post-merge sweep) ‚Äî ERC-777 `operatorSend(address,address,uint256,bytes,bytes)` has identical drain semantics to `transferFrom` and was outside the classifier's ruleset entirely.
- **#757 / #760** (this design's trigger, found by SEC's 28-agent adversarial fan-out while adjudicating #737) ‚Äî the same substrate, one layer up: it is not the `custom_call` classifier that misses the recipient, it is `assertTransactionSafe` block 5 itself, for **every** recognized protocol destination, not just `custom_call`-reachable selectors. #757 is selector-membership-without-argument-check; #760 (LiFi Diamond, `allowedAbi: null`) is the same gap with no selector check at all.

The recurring pattern across all four: a gate keys on **which selector/rule matched**, and treats "selector recognized" as sufficient without ever decoding **which argument carries the recipient**. #759 is ARCH's design to close this at its true locus ‚Äî the pre-sign argument level ‚Äî rather than patching the next selector that surfaces it.

## 3. Design

The core idea ‚Äî argument-level recipient/authority authorization at the pre-sign chokepoint ‚Äî survived all three review rounds unchanged; every requested change lands inside the mechanism, not against the idea. The mechanism was revised three times: v1 was an inclusion map (fails open on any unmapped selector, reproducing #757 in miniature); v2 inverted to a fail-closed exemption model; v3 folded round-2's one new blocker (a contact predicate that admitted unsigned entries) plus mechanical residuals; **v4 folds round-3's authoritative SEC + REVIEW verdict** ‚Äî most substantially, narrowing the hard gate to wallet-only and deferring the signed-contact leg to a follow-on issue (D4), plus corrections at D1, D2, D2-rot, D3, D7, D8, D9, and ¬ß5. What follows is the fully consolidated v4 design ‚Äî not a base plus a changelog.

### D1 ‚Äî Locus: `runEvmPreSignGuards`, after a fail-closed device-account match

The check does **not** live inside `assertTransactionSafe`. That function receives only `tx.from`, which is agent-supplied (`buildCustomCall` sets `from: p.wallet`, and `p.wallet` is `args.wallet` off the tool call ‚Äî nothing pins it to the connected device before `assertTransactionSafe` runs). Resolving a recipient against `tx.from` at that point is resolving against a value the attacker also controls: set `wallet = to = ATTACKER`; if the connected WalletConnect session is in fact still bound to the real victim's device ‚Äî an assumption this check does not itself verify ‚Äî the signature request would still target the victim's account while `recipient === tx.from` passes, because nothing at this point has confirmed `tx.from` against the device at all.

The check instead runs in `runEvmPreSignGuards` (`execution/index.ts`, evidence-pointer L1819 at recon), **after** the existing device-account match (evidence-pointer L1842-1846 in the recon commit ‚Äî re-resolve by reading the account-match call site relative to `runEvmPreSignGuards`'s own body post-decomposition). At that point `tx.from` has been validated `== connected device account`, so comparing a decoded recipient to `tx.from` is sound ‚Äî subject to the precondition below being fixed; it is not fixed today.

**Precondition ‚Äî the account-match itself must be fail-closed on BOTH of its known fail-open branches, as a SAME-PR co-merge invariant with the recipient check.** The match fails open two independent ways today: (1) `getConnectedAccounts()` returning an empty array (no paired session) skips the match entirely rather than refusing; (2) the guard is itself gated `if (tx.from && !isDemoMode())` (evidence-pointer `execution/index.ts:1842`) ‚Äî a falsy or missing `tx.from` skips the match on its own, independent of (1), because `from` is optional on `UnsignedTx` and some send paths substitute `getConnectedAccounts()[0]` when it is absent. A real-signing, non-demo path hitting EITHER branch must **refuse**, not silently skip the comparison and fall through to an unvalidated `tx.from`. Demo mode (simulated, no real signature ever produced) is exempt from both. **This precondition does not ship separately from the recipient check ‚Äî the two land in the same PR**, because the recipient check is unsound without it (¬ß4 carries falsifiers for both fail-open branches).

**Ack-independence, not code-ordering, is the bypass-proof.** The recipient check's non-bypassability comes from the check being **structurally independent of the ack flags** ‚Äî it never reads `acknowledgedNonProtocolTarget`, `safeTxOrigin`, or `acknowledgedNonAllowlistedSpender` ‚Äî not from running after `pre-sign-check.ts:336` in source order (it runs in a different function, `runEvmPreSignGuards`, not `assertTransactionSafe`). Falsifier: an ack-stamped call to a recognized destination with an unauthorized recipient must still REFUSE.

**Enforcement point.** `runEvmPreSignGuards` is invoked only from `previewSend`; `sendTransaction` never re-runs it, inheriting the guard only through the mandatory-preview gas-pin + `previewToken` binding (ARCHITECTURE.md ¬ß3.3). Soundness of that inheritance is covered in ¬ß5 below ‚Äî the relevant tx-store freeze work is landed, and ¬ß5's send-time recipient re-check is the primary enforcement for the one piece the freeze does not reach (the connected-account set).

**Export requirement.** `classifyDestination` is currently module-private to `pre-sign-check.ts` (defined at line 105). It must be exported (or its recipient-relevant logic re-derived) so `runEvmPreSignGuards` can classify the destination independently. `dest === null` (unrecognized) defers entirely to D9 ‚Äî no hard gate; an ack-stamped unknown-destination call escapes this seam by design, unchanged from today.

**Decode source: block 5's accept-set union, never `dest.allowedAbi` alone.** Block 5's real accepted-selector set for a given `dest.kind` is, in several cases, a UNION across ABIs ‚Äî not the single curated `allowedAbi` field alone. Sharply for `weth9`: `dest.allowedAbi` is `erc20Abi`, but block 5's own acceptance test for that kind is the union `ERC20 ‚à™ wethAbi` selectors (`pre-sign-check.ts`'s per-kind selector-set construction, ~L362-396). This union diverges from the single curated `allowedAbi` field on four destination kinds ‚Äî `weth9` is the one this design has concretely traced; the remaining three are for DEV to re-derive against the live per-kind switch, since this doc does not re-walk it kind-by-kind. D2's decode-and-enumerate step, and D2-rot's static enumeration, must both walk the SAME union block 5 actually accepts for a `dest.kind` ‚Äî either re-deriving that union locally, or routing through the already-exported `decodeCalldata` (`signing/decode-calldata.ts:545`, union construction ~L61), which is understood to already resolve this for the signing/verification path. Using `dest.allowedAbi` alone as the decode source silently under-covers the diverging kinds: a legitimate call on a selector present only in the wider union would either throw against the narrower curated ABI (an over-block ‚Äî D2-rot's top-level-decode-refuse firing on a VALID call) or decode against the wrong shape and misclassify an argument.

### D2 ‚Äî Fail-closed authorization gate over every address-typed argument

Replaces v1's inclusion map. For every recognized-destination selector that passes block 5, decode the call and enumerate **every address-typed argument path**, including nested tuples and arrays-of-tuples (e.g. Uniswap's `ExactInputSingleParams.recipient`, Morpho's `MarketParams` tuple, Permit2's `AllowanceTransferDetails[]`). Each such address must fall into exactly one of four buckets:

1. **NON_RECIPIENT_ALLOWLIST** ‚Äî admission is a **stated test**, not a name-list: an address argument qualifies only if **it cannot cause the caller's value to be spent to a third party's benefit, and cannot confer authority on a third party.** Per-entry evidence (replaces the prior bare name-list):
   - Lido `submit.referral` ‚Äî a fee-routing address; the referral fee is paid from the protocol's own emission, never redirected from the caller's principal or output.
   - EigenLayer `depositIntoStrategy.strategy` ‚Äî the strategy contract itself, not a beneficiary; the depositor's own address remains the position holder.
   - Every Morpho `marketParams.{loanToken, collateralToken, oracle, irm}` ‚Äî venue/infrastructure addresses fixed by the market's own identity, never a redirectable beneficiary (D2-rot below; the `oracle`/`irm` pair is the easy-to-miss half of this tuple).
   - Aave `borrow.onBehalfOf` ‚Äî passes the test by **direction**, not by absence of a third party: `borrow`'s ABI (`aave-pool.ts:42-52`) carries no separate receiver parameter, so the borrowed asset always transfers to the caller (`msg.sender`) regardless of `onBehalfOf`; `onBehalfOf` only designates which account's debt is charged, and charging a third party's account requires that party's own prior on-chain credit-delegation approval. The caller's own value is never spent to a third party's benefit here.
   - **Aave `repay.onBehalfOf` and Morpho `repay.onBehalf` do NOT belong in this bucket ‚Äî moved to D3's hard-gated list, below.** `repay` is the mirror image of `borrow`: it spends the CALLER's own tokens to reduce a THIRD PARTY's debt. Setting the argument to an attacker's address pays down the attacker's obligation using the victim's funds ‚Äî a real drain the general test above correctly excludes. A prior revision exempted it in error by pattern-matching on the shared `onBehalfOf` argument name rather than checking the direction of value flow.
2. **Wallet-only (D4)** ‚Äî resolves to the connected device wallet (`tx.from`, device-verified per D1). The signed-contact leg is **deferred to a follow-on issue**; see D4.
3. **Authority designation (D6)** ‚Äî an authorization-conferring argument (spender/operator), handled by the unified authority policy, not the recipient policy.
4. **USER_DIRECTED_RECIPIENT** ‚Äî a fourth bucket (below) for arguments the tool's whole purpose is to route to an address the *user*, not the protocol, names ‚Äî pass-through at this gate, governed by the existing resolver+device-display defense instead, and conditioned on the absence of a server-populated provenance stamp ‚Äî see D3.

**Any address argument that does not classify into one of these four buckets ‚Üí REFUSE.** Adding a new ABI thus fails toward a human-clearable refusal, never a silent drain ‚Äî the opposite failure mode from v1's inclusion map and from #757 itself.

**The fourth bucket, and why it must exist (reconciling D2 with D3).** `erc20Abi.transfer.to` (and the stETH/WETH9 `transfer` unions) is a user-directed recipient by the tool's own design ‚Äî `prepare_token_send` exists precisely to let the user name a fresh address, and that flow is governed by `resolveRecipient` labeling plus the on-device address render (¬ßD3), not a pre-sign hard refusal. Without a fourth bucket, `transfer.to` fits none of buckets 1-3: bucketing it as a recipient (2) reopens the over-block this design deliberately removes (¬ßD3); leaving it unbucketed fails the anti-rot enumerator below, which throws on any unclassified path. The fourth bucket is what lets the module-load enumerator pass while runtime still treats `transfer.to` as PASS, defended by the resolver/device-render layer ‚Äî D2 and D3 are explicitly reconciled on this argument, not left in tension.

### D2-rot ‚Äî Module-load anti-rot enumeration (the completeness mechanism)

A hand-seeded classification map rots the moment a new ABI or a new selector on an existing ABI is recognized without a matching entry ‚Äî exactly the #757 failure mode, recreated one layer down. The anti-rot mechanism: **at module load**, recursively enumerate every address-typed argument path across every recognized ABI and assert each path is a member of `NON_RECIPIENT_ALLOWLIST ‚à™ RECIPIENT_MAP ‚à™ AUTHORITY_MAP ‚à™ USER_DIRECTED_MAP`. **Throw at startup / fail a unit test red** on any unclassified path ‚Äî this converts "new recognized ABI silently ungated" into "won't boot," a machine-checkable falsifier rather than a hand-maintained completeness obligation.

**Scope note: the enumerator keys on ABIs; the destination-recognition set keys on CONTRACTS.** These are different domains ‚Äî `classifyDestination` recognizes a destination by its `to` address (a `CONTRACTS`-keyed lookup) and attaches an ABI to it; the enumerator walks the recognized *ABIs* directly. Keeping the two domains distinct in the design's own vocabulary matters for DEV: an enumerator that silently drifted to iterate a separately-maintained ABI list, rather than the exact set `classifyDestination` attaches per contract, would reopen a rot path this section exists to close.

**The enumerator's ABI iteration domain must be DERIVED from `classifyDestination` itself, not a separately hand-maintained list ‚Äî this complements, not replaces, the scope note above.** `classifyDestination` (`pre-sign-check.ts:104-190`) and block 5's kind‚Üíselector switch already define, per destination kind, exactly which ABI is recognized. The enumerator must walk that same computed set, not a second array the enumerator owns independently. A separately hand-listed enumerator domain is itself a rot vector structurally identical to the one this section exists to close: a future PR adds a new kind + ABI to `classifyDestination` (and to block 5's switch) without remembering to add the same ABI to the enumerator's own separate list ‚Äî the server boots clean, the new kind's arguments are never enumerated, and the "won't boot on an unclassified path" guarantee never fires, because the new ABI was never in the enumerator's domain to begin with. This reopens the #757 failure mode one level up from where D2-rot was built to close it. **Boot falsifier:** assert the enumerator's ABI domain is set-equal to the ABI set `classifyDestination`/block 5 actually recognize; fail boot if they diverge in either direction (an ABI present in one set but not the other).

Five scoping/state corrections, all load-bearing for the enumerator to be buildable and complete:

- **Scope to state-mutating function inputs only ‚Äî a semantics exclusion, not a volume-management shortcut.** Restrict enumeration to `stateMutability ‚àà {nonpayable, payable}` function **inputs**; exclude `view`/`pure` functions and all address-typed **outputs** (e.g. Comet `getAssetInfo`'s output, Aave `getUserAccountData`'s output tuple, Uniswap `positions()` outputs, ERC-20 `balanceOf`/`allowance` reads). The reason is not that walking read-only surface is impractically large ‚Äî it is that a read-only address categorically has no home in ANY of the four buckets (it is not a recipient, not an authority, not user-directed; it is not writable state at all), so including it would fail boot BY CONSTRUCTION on every single one, not as a matter of degree. Scoping to state-mutating inputs is a semantic exclusion of a category that cannot be classified, not a shortcut to keep the list short.
- **Fail-closed on opaque byte containers, not just typed addresses.** The enumerator as stated above only walks statically-typed `address` arguments. A recipient can be smuggled inside a `bytes`/`bytes32` argument on a state-mutating function, invisible to that static walk. Any state-mutating function carrying a `bytes`/`bytes32` parameter is **unclassified-unless-explicitly-exempted** ‚Äî it must be on an explicit MUST-DECODE / MUST-RECURSE / REQUIRE-EMPTY list (below) or the module fails to boot.
- **A third state ‚Äî REQUIRE-EMPTY ‚Äî for opaque bytes params with no available decoder.** Five currently-recognized state-mutating functions carry a `bytes`/`bytes[]` parameter: Morpho `supply`/`repay`/`supplyCollateral.data` (`morpho-blue.ts:65/113/128` ‚Äî each an arbitrary borrower-callback payload with no decoder available to this design) and the two Uniswap `multicall(bytes[])` entries (one per ABI; D7 below, MUST-RECURSE). Morpho's `data` param is **REQUIRE-EMPTY**: refuse the call unless the value is zero-length. This is stricter than a blanket exemption ‚Äî it does not claim to have proven the callback payload can never carry a recipient, it simply refuses to accept a non-empty one it cannot decode, closing the risk without asserting more than is known.
- **Universal Router example relabeled FORWARD-LOOKING ‚Äî it was presented as a live gap and is not one.** Uniswap's Universal Router `execute(bytes commands, bytes[] inputs)` is **not currently a recognized destination** ‚Äî `classifyDestination` recognizes only the NFT position manager and SwapRouter02 ‚Äî so a Universal Router call resolves `dest === null` and is handled by block 4's catch-all, never reaching block 5 or this enumerator at all. A prior revision cited it as a live example of the bytes-container escape; it is not one today. It remains a useful **forward-looking** illustration of the general risk class (zero top-level address arguments, every recipient inside an opaque blob) that this MUST-DECODE/REQUIRE-EMPTY discipline must also catch the day Universal Router, or anything shaped like it, becomes a recognized destination. The two Uniswap `multicall(bytes[])` entries (D7) are the real, currently-recognized bytes-container path this design closes today.
- **The top-level decode failure also refuses.** Mirror the existing approve-decode catch (`pre-sign-check.ts:262-266`): a top-level `decodeFunctionData` throw on a recognized-destination call is a REFUSE, the same posture already specified for multicall sub-calls (D7) but stated explicitly at the top level too.

A structural falsifier accompanies the enumerator itself: a test asserting it finds `.recipient` nested inside `exactInputSingle`'s tuple parameter, proving the tuple/array-of-tuple walk actually works, not just that it compiles.

### D3 ‚Äî Scope of the hard gate: protocol-embedded recipients, not user-directed sends

The **wallet-only** hard gate (non-ack-bypassable ‚Äî see D4) applies only to **protocol-embedded recipient arguments** ‚Äî arguments where the `prepare_*` tool sets the value to the wallet address by construction, so any other value is a genuine anomaly, never a legitimate call shape the tool itself would produce:

- Aave `withdraw.to`
- Morpho `withdraw`/`borrow`/`withdrawCollateral.receiver`
- Uniswap tuple `exactInput`/`exactOutput.recipient` (swap-class), `unwrapWETH9.recipient` (**not** `sweepToken` ‚Äî absent from `swapRouter02Abi`, so unreachable by decode; already refused via D7's absent-selector-throw rule, not by being on this list)
- Lido `requestWithdrawals.owner`
- Permit2 `transferFrom.to` and batch `.to[]`
- ERC-4626 `deposit`/`mint`/`withdraw`/`redeem.receiver`
- Self-funded-credit arguments: Aave `supply.onBehalfOf`, Morpho `supply`/`supplyCollateral.onBehalf` ‚Äî these are real drains (an attacker-controlled `onBehalf` deposits the user's funds into the attacker's position, withdrawable later by the attacker) even though the *transfer* direction is inbound, because the position/aToken lands at the named address, not the caller.
- **Aave `repay.onBehalfOf`, Morpho `repay.onBehalf` (moved from D2's bucket 1 ‚Äî see D2's corrected stated test).** A repay call spends the CALLER's own tokens to reduce a THIRD PARTY's debt; setting the argument to an attacker's address pays down the attacker's obligation using the victim's funds ‚Äî the mirror-image drain of the self-funded-credit row above, on the repayment side rather than the deposit side.

**Also dropped from the hard gate: Uniswap `mint.recipient` and `collect.recipient` ‚Äî moved to bucket 4 (USER_DIRECTED_RECIPIENT), not hard-gated.** These are user-choosable on the LIVE standalone tools: `prepare_uniswap_v3_mint` and `prepare_uniswap_v3_collect` (`src/index.ts:4613/4631/4679`) build `recipient = p.recipient ?? p.wallet` (`lp/uniswap-v3/actions.ts:190,827`) ‚Äî a user legitimately mints an LP position, or collects fees, to a fresh address they name, exactly as with `erc20 transfer.to`. Hard-gating them wallet-only would refuse that legitimate flow; a prior revision's "prepare_* sets recipient=wallet by construction" premise did not hold for these two. They inherit bucket 4's full mechanism, not an unconditional pass: governed by the resolver+device-render defense, AND conditioned on the absence of a server-populated provenance stamp (same as `transfer.to`, above) ‚Äî a `prepare_custom_call` call to Uniswap's position manager stamping `acknowledgedNonProtocolTarget` on a `mint`/`collect` with a non-wallet recipient does NOT qualify for bucket 4's pass-through and is evaluated as hard-gated instead, closing the same #757-shaped exploit on this protocol. The REBALANCE-multicall path (`lp/uniswap-v3/actions.ts:1120,1175`), which hardcodes `recipient=wallet`, is unaffected ‚Äî it continues to pass under D7's universal-per-leg rule regardless of which bucket `mint`/`collect.recipient` resolves to, because that flow's recipient is the wallet either way.

**Dropped from the hard gate: ERC-20/native `transfer`/send.** A hard refusal on `prepare_token_send` to a fresh, unsaved address contradicts the tool's entire purpose ‚Äî sending to an address the user names for the first time is the normal case, and block 1 already lets a **native** send to any address through on the strength of the on-device recipient display alone; refusing the equivalent ERC-20 `transfer` at this gate is an inconsistency the pre-sign layer cannot resolve on calldata alone (that distinguishing signal ‚Äî which tool built the call, and what the user was shown ‚Äî lives above this gate).

**A prior revision's basis for this was FALSE and is corrected, not restated.** The prior text asserted "`transfer`-via-`custom_call` is already closed by the #741-family classifier" and that a fresh address is "resolver-labeled." Neither holds. `transfer(address,uint256)` (`0xa9059cbb`) cannot structurally be a #741-family gate: that gate's rule shape requires a `from`/`owner` argument to compare against the wallet (the #711/#737/#741 pattern), which bare `transfer` does not carry. The custom-call classifier's own `transfer` rule (the #652 rule) is itself ack-bypassable via an optional `acknowledgeKnownExfilPattern` flag, not a non-bypassable gate. And a fresh, never-saved address carries no resolver label at all ‚Äî "resolver-labeled" does not describe the exact case this bucket exists for.

**Corrected fix: bucket 4 (USER_DIRECTED_RECIPIENT) is conditioned on the ABSENCE of a server-populated provenance stamp ‚Äî stated as "no server-populated recipient block," not on an assumed upstream gate.** A `transfer.to` argument reaching this seam through `prepare_custom_call` carries `acknowledgedNonProtocolTarget`, a flag stamped by the SERVER (not the agent) on the custom-call path (`execution/index.ts`, evidence-pointer L1724 ‚Äî one of three stamp sites total, alongside the Curve builder's own stamp and one further site; ARCHITECTURE.md ¬ß3.2 currently documents two, the third is a correction tracked under the separate #762 Curve mitigation track, not this design). A call carrying that stamp does NOT qualify for bucket 4's pass-through and must instead be evaluated as a protocol-embedded/hard-gated argument, or refused if unclassifiable. A `transfer.to` with **no** such stamp ‚Äî i.e. built by `prepare_token_send`, which never sets it ‚Äî qualifies for bucket 4's pass-through.

**UNVERIFIED ‚Äî flagged, not adopted as a closed question.** Whether an agent can set or inject `acknowledgedNonProtocolTarget` through some path OTHER than the sanctioned server-side stamp sites is not established here: `from` is optional on the caller-visible `UnsignedTx` shape, which raises the general question of whether any field on that shape can be agent-substituted through some other handle-issuing route before reaching this check. **DEV must trace every path capable of setting this flag before adopting the provenance-stamp condition as bucket 4's gate** ‚Äî if an agent-reachable path to set the stamp exists outside the known server-side sites, this condition does not hold as specified and needs redesign, not adoption as-is.

**Residual, stated honestly ‚Äî not claimed defended by a gate that doesn't exist.** `prepare_token_send(to=ATTACKER)` is defended ONLY by the on-device address render ‚Äî there is no cryptographic or provenance-based backstop at this pre-sign seam for that specific call shape. This is the accepted, named residual of dropping the hard gate here (the usability rationale above), not an oversight papered over by a false claim of coverage.

### D4 ‚Äî WALLET-ONLY hard-gate predicate (signed-contact leg DEFERRED)

**A prior revision's wallet-or-signed-contact predicate is superseded. The hard gate's predicate is now `recipient === tx.from` (the device-verified wallet, per D1) ‚Äî ONLY.** Round-3 SEC review found the signed-contact leg's trust root unsound at its foundation, not merely under-specified: `verifyEvmBlob` recovers the signature against `blob.anchorAddress` (`contacts/verify.ts:211`, `return derived === blob.anchorAddress`) ‚Äî a field carried *inside the signer's own payload*, not the device's independently-known address ‚Äî and `reverseLookup` calls `verifyEvmBlob` directly, with no call anywhere in `src/contacts/` to `getConnectedAccounts()`. An attacker can self-sign a blob whose `anchorAddress` is the attacker's own choice, producing a clean `state: "match"` verdict with no device involvement at all.

**A prior revision's claim here was false and is removed, not softened:** it asserted "`verifyEvmBlob` recovers the signer against the device's own EVM address." That is not what the code does ‚Äî it recovers the signer against `blob.anchorAddress`, a self-referential, attacker-choosable field.

**Decision: the hard gate accepts ONLY `recipient === device-verified tx.from`.** This costs nothing functionally today: every current protocol builder hardcodes its recipient/receiver/onBehalf argument to the wallet address, so removing the (unsound) contact leg refuses nothing any live tool produces ‚Äî it only removes a trust root that had no legitimate exercising path to begin with. This is simpler, aligned with the design's own simplify mandate, and unblocks #757/#760 on a trust root that is sound as specified, rather than gating the incident fix on a device-binding invariant that does not yet hold.

**The signed-contact leg is DEFERRED to a follow-on issue, not dropped outright.** It remains the right direction once its prerequisites land: (1) a real device-anchor binding for the contact blob ‚Äî the actual fix to the defect above, anchoring the address to something the device confirms independently rather than a self-asserted field inside the signed payload; (2) a single-entry-delta approval model (approving the one changed/added entry, not a whole-blob signature that also re-authorizes every unchanged entry alongside it); (3) a parallel device-binding treatment for the Bitcoin contact-signing path (preimage/sha256 handling). File this as a tracked follow-on issue before DEV implements #759 from this document.

**One invariant from the contact-signing surface stays verified true and remains load-bearing even under wallet-only:** unsigned contact entries cannot be laundered into a signed blob. `addContact`'s signed-path branch builds its `oldEntries` list exclusively from the on-disk **signed** blob (`readContactsFile()` ‚Üí `file.chains[chain].entries`, `contacts/index.ts:300,321`) ‚Äî the unsigned, agent-mintable in-memory store (the separate no-Ledger-paired branch, `contacts/index.ts` ~L280-295) is never read by the signed-path code at all, so an unsigned entry cannot ride into a subsequently-signed blob. This does not rescue the contact leg for this increment ‚Äî it is deferred regardless, on the anchor-binding defect above ‚Äî but it rules out one additional laundering path SEC checked for and confirmed closed.

**Tamper handling for user-directed sends is unchanged.** For user-directed sends (¬ßD3, not hard-gated), a tampered contact entry stays non-fatal warn-and-proceed ‚Äî the existing prepare-time behavior; this design does not touch that path.

**Residual, not a #759 regression ‚Äî the whole-blob signing model itself.** Independent of the anchor-binding defect above (which blocks the contact leg entirely for this increment), the signed device prompt shows the **entire** `VaultPilot-contact-v1:{JSON}` blob (`signers/contacts/evm.ts:5-14`), not a per-address confirmation screen ‚Äî "signed" means "the device holder approved this whole blob," not "confirmed this specific address." This is a property of the follow-on's design space, not #759's, and motivates one of the three prerequisites named above (the single-entry-delta approval item).

### D5 ‚Äî Permit2: exact-selector scope, not a broad ABI grant

Add Permit2 to the recognized-destination set with an ABI scoped to **transfer selectors only**: `AllowanceTransfer.transferFrom(address,address,uint160,address)` (`0x36c78516`, recipient = `args[1]`) and its batch form (`transferFrom(AllowanceTransferDetails[])`, recipient = each element's `.to`; verify the batch selector hex against viem before landing ‚Äî still unverified). `approve`/`permit` (authority grants) are **excluded** from this ABI grant and instead route through D6 (the unified authority policy) ‚Äî admitting them here would open an authority grant ack-free. **The recognized ABI must be exact-selector, not the full Permit2 interface**: `SignatureTransfer.permitTransferFrom`/its batch form (`transferDetails.to`, tuple/array-nested) and `lockdown` must either be mapped with their own nested `.to` paths or remain outside the recognized ABI so block 5 fail-closes them ‚Äî do not admit a broader Permit2 ABI than these named selectors.

### D6 ‚Äî Unified authority-designation policy

The gate covers value/credit recipients **and** authority designations ‚Äî an address argument that confers spend/transfer authority without itself receiving anything is drain-equivalent and must be covered by the same policy: ERC-20 `approve` spender, ERC-721/1155 `setApprovalForAll` operator, Permit2 `approve`/`permit` spender. This reconciles with block 2's existing `approve` spender-allowlist (`pre-sign-check.ts:216-280`) ‚Äî authority designation is **one** policy consumed from both loci (see D10), not two independently-maintained lists that can drift apart.

### D7 ‚Äî Multicall: universal per-leg, bounded, cached

`multicall(bytes[])` (`0xac9650d8`, present on both Uniswap ABIs) requires four properties together, not any one alone:

1. **Universal per-leg invariant.** Decode every sub-call and apply D2 to **every leg**, not an existential "does *some* later leg release to the wallet" check ‚Äî that shape is launderable: `multicall([exactInputSingle(recipient=router), exactInputSingle(A‚ÜíB, recipient=ATTACKER), unwrapWETH9(min,wallet)])` satisfies "a later leg releases to wallet" while draining through the unchecked middle leg.
2. **Router-self exception ‚Äî conjunctive and asset-bound (was an existential, exploitable disjunction).** A swap-leg `recipient == the exact chain SwapRouter address` is permitted ONLY when BOTH hold: **(a)** that leg's `tokenOut == WETH` (a router-self recipient is legitimate only as the intermediate stop before an unwrap, never for an arbitrary output token), AND **(b)** the SAME multicall contains a sibling `unwrapWETH9(amountMinimum > 0, recipient)` leg whose `recipient` independently passes D2. **`refundETH()` does NOT satisfy requirement (b)** ‚Äî it takes no arguments and releases native ETH unconditionally to `msg.sender`, so it cannot serve as the D2-checked terminal leg the coexistence requirement demands; a router-self leg paired only with `refundETH()`, with no `unwrapWETH9` present, REFUSES. Without BOTH (a) and (b) present together, a router-self recipient on any leg REFUSES ‚Äî this closes the prior existential hole, where a non-WETH swap leg with `recipient == router` could ride through on an unrelated `unwrapWETH9`/`refundETH` leg elsewhere in the same call that had nothing to do with that leg's own output token. **The terminal release leg itself is WALLET-ONLY** (consistent with D4 ‚Äî not wallet-or-contact), and never sanitizes sibling legs regardless of how many legs use the router-self exception.
3. **Fail-closed on unrecognized sub-call selectors.** Re-validate every sub-call selector against block 5's `allowedSelectors` for the destination ABI; a selector absent from that ABI (e.g. SwapRouter02's `sweepToken`/`sweepTokenWithFee`/`unwrapWETH9WithFee`, which are absent from `swapRouter02Abi`) must throw on decode and REFUSE, never silently fall through as unchecked.
4. **Total decoded-sub-call budget (breadth-bound, not depth-bound) and single contact-blob resolution.** Refuse beyond a small fixed TOTAL count of decoded sub-calls across all recursion depths combined ‚Äî a shallow-but-wide call (e.g. one `multicall` with many sibling top-level legs) needs the same bound a deep one would, so the budget is on total decoded sub-calls, not depth (bounded-loop canon, never "recurse or refuse unboundedly" on depth alone). Resolve the signed-contacts blob **once** per `assertTransactionSafe`/gate invocation and thread it through every leg's check, rather than re-reading disk and re-running `verifyEvmBlob` once per leg. (Note: under D4's wallet-only hard gate, no contact-blob resolution occurs inside this check for the current increment ‚Äî this bullet states the design for when the deferred contact leg, D4, lands in its follow-on, so the caching discipline is specified once rather than reintroduced later.)

### D8 ‚Äî LiFi: gate the extractable receiver, ack-gate (fail-closed) the remainder

LiFi Diamond is `allowedAbi: null` and therefore block-5-exempt (ARCHITECTURE.md ¬ß3.4) ‚Äî but it still carries `_receiver` in bridge/swap calldata that the signing layer already decodes and discards (via the already-imported `lifiDiamondAbi` + `tryDecodeLifiBridgeData`). Enumerate the LiFi route shapes whose `_receiver` those decoders can extract, and apply the wallet-only invariant (D4) to it on those shapes. For any LiFi route shape where receiver extraction is not feasible, that shape is downgraded to **ack-gated and fail-closed** with `_receiver` surfaced to the user ‚Äî never silently exempted the way it is today. Confirm the generic-swap and bridge decoders cover every route facet `prepare_swap` itself can emit, so no route the MCP's own tool produces is silently downgraded to the ack path for lack of decoder coverage.

**Named exemption ‚Äî `NON_EVM_RECEIVER_SENTINEL` (non-EVM bridge destinations).** When a LiFi bridge route's destination chain is non-EVM (`toChain ‚àà {solana, tron}`), the bridge calldata's `receiver` field is required to carry a fixed protocol sentinel value, not an EVM address (`swap/index.ts:199-202` ‚Äî the live code itself throws if the decoded receiver does not exactly match the expected sentinel for that destination chain). Applying the wallet-only invariant to this field literally would REFUSE every non-EVM bridge route ‚Äî a functional break, not a security improvement: the sentinel carries no attacker-redirectable value, and the actual non-EVM destination address is bound elsewhere in the route, outside this seam's EVM-address model. `NON_EVM_RECEIVER_SENTINEL` is therefore its own named exemption ‚Äî structurally similar to bucket 1 (NON_RECIPIENT_ALLOWLIST) but scoped to this one exact sentinel value on non-EVM-destination bridge routes; it is neither compared against `tx.from` (not wallet-only) nor treated as an extractable user/protocol receiver (not D8's main rule) ‚Äî it is its own recognized, address-shaped, non-recipient constant.

Falsifier per branch: an EVM-destination LiFi bridge route with `_receiver != wallet` ‚Üí REFUSES (D8's main rule, unchanged). A non-EVM-destination LiFi bridge route carrying the exact expected sentinel value ‚Üí PASSES (this exemption). A non-EVM-destination route whose receiver field does NOT match the expected sentinel ‚Üí REFUSES (the exemption is exact-value, matching the live code's own throw-on-mismatch behavior, not a blanket non-EVM pass).

This closes only the recipient dimension of LiFi's exposure. The broader arbitrary-calldata-to-LiFi-Diamond bypass ‚Äî `prepare_custom_call` reaching the same address with arbitrary calldata and arbitrary native value, no selector check at all ‚Äî is ARCHITECTURE.md ¬ß3.4, ruled **UNSOUND ‚Äî REQUIRES MITIGATION** by SEC (#760) as a wider, separate mitigation thread. #759 does not close #760; it closes the recipient-visibility gap on the routes LiFi's own decoders can read.

### D9 ‚Äî Explicit residual scope: what this seam does not see

State plainly, not imply coverage: a top-level-argument seam cannot see **nested** calldata, and an address-authorization gate does not itself evaluate asset quality or price on the path it authorizes. Named gaps stay open, owned by separate mitigation threads or stated as an explicit design boundary:

- **Safe `execTransaction`** (ARCHITECTURE.md ¬ß3.5) ‚Äî the outer call carries `safeTxOrigin: true` and skips block 4, but the inner `SafeTx` body (`to`/`data`, potentially `operation === 1` DELEGATECALL) is never decoded by anything. SEC ruled this UNSOUND ‚Äî REQUIRES MITIGATION.
- **LiFi `SwapData.callTo`** nested inside a struct, beyond what D8's receiver extraction covers.
- **The ASSET dimension ‚Äî not a nested-calldata gap, a structural boundary of what this gate claims to do.** D2's gate validates WHERE an argument points; it says nothing about WHAT is transacted or at what price. `exactInputSingle({tokenOut: ATTACKER_DEPLOYED_TOKEN, recipient: tx.from, amountOutMinimum: 0})` passes this design's checks in full ‚Äî the recipient is the honest, device-verified wallet ‚Äî while the wallet receives an attacker-deployed worthless token at an unbounded price. **D2's "never a silent drain" authorizes the ADDRESS, not the ASSET.** Asset/price quality is the swap-class prepare-time tools' job (min-out math, quoter enforcement); `prepare_custom_call` bypasses that protection entirely, exactly as it bypasses every other prepare-time invariant (repo CLAUDE.md "Per-protocol `prepare_*` vs. `prepare_custom_call` cutoff").

The first two are separate mitigation threads (see ¬ß5's explicit scope boundary for their tracking numbers); the third is not covered by any thread because it is out of this gate's claimed scope by design, not a gap in its execution. The design artifact states all three explicitly so a reader does not assume the argument-authorization gate reaches everywhere a `to`/`data` pair can appear, or that authorizing a recipient says anything about what that recipient receives.

### D10 ‚Äî Single-source the recipient/authority map across both loci

The #741-family prepare-time custom-call classifier and this pre-sign gate must consume **one** recipient/authority map (one derived from the other, not two independently maintained). **Note:** the divergence this section originally reconciled ‚Äî prepare-time wallet-only vs. pre-sign wallet-or-signed-contact ‚Äî does not exist in this increment, because D4's wallet-only revision makes the pre-sign hard gate wallet-only too; the two loci are, for now, identical on the recipient predicate. Single-sourcing the map remains the design going forward regardless: it is what keeps the two loci's selector/argument lists from drifting apart over time on everything OTHER than the wallet-vs-contact predicate (new ABIs, new authority-designation arguments, the C-list), and it is what will let the deferred signed-contact leg (D4's follow-on) land in ONE place both loci consume when it ships, rather than reopening exactly the drift risk this section exists to prevent.

## 5. Dependencies & sequencing

**#751 is not a #759 dependency.** `safe-tx-store.ts`'s `lookupSafeTx` (#751) is off-path for this gate ‚Äî it is consumed only by `safe/actions.ts` and `safe/execute.ts` (`execution/index.ts` never calls it), and D9 already names the Safe inner-body decode as a residual this seam does not cover regardless of #751's state. #759's gate runs in `runEvmPreSignGuards`, consuming the EVM tx-store object that `issueHandles` deep-freezes (`tx-store.ts:178`) ‚Äî that freeze work (`#710`/`#742`) is **landed**, and `sendTransaction` re-reads the SAME frozen object `preview_send` checked, so the two cannot diverge on the frozen tx's fields. **The bytes-level dependency this design rests on ‚Äî the tx-store deep-freeze ‚Äî is already landed and satisfied; no open freeze issue gates #759.**

**The send-time recipient re-check is PRIMARY enforcement, not a recommended alternative.** The actual soundness gap here is narrower than a tx-store question: the recipient check's INPUT ‚Äî the connected-accounts set (`getConnectedAccounts()`, backed by the module-mutable current WalletConnect session) ‚Äî lives in no tx-store and is not covered by the tx-object freeze at all. A frozen tx does not freeze the WC session it is checked against; a session change between `preview_send` and `send_transaction` (a pairing drop, an account switch) is the real divergence risk, not tx-object mutation. The fix is a re-check at send time: inside `sendTransaction`, AFTER `consumeHandle` (evidence-pointer `execution/index.ts:2272`) and BEFORE `requestSendTransaction` (evidence-pointer `execution/index.ts:2284`), re-decode the tx object about to be forwarded and re-compare its recipient against a freshly-read `getConnectedAccounts()`; fail-closed on an empty account set (the same fail-closed precondition as D1). The `preview_send`-time run of this check is retained but relabeled **UX-only** ‚Äî part of the `CHECKS PERFORMED` display the user sees before confirming, explicitly NOT the enforcement gate. **The send-time re-check is the gate.**

Falsifiers: (i) swap the connected-account set between `preview_send` and `send_transaction` on the same handle ‚Üí `send_transaction` REFUSES; (ii) `preview_send` ‚Üí `preview_send` (re-preview) ‚Üí `send_transaction` still evaluates the recipient check AT SEND, never carrying forward an earlier preview's verdict. **Do not argue this adds a "zero evaluations" cost concern** ‚Äî the gas-pin guard already re-runs exactly once per pin today at the same call site; one more re-decode-and-compare there is the same shape of cost the pin guard already pays, not a new class of overhead.

**Relation to #756.** #756 (recipient-verified send path) is the legitimate feature this seam's refusal message should point users toward for a genuinely new non-wallet recipient on a protocol-embedded argument ‚Äî it is the durable-binding alternative to "just widen the hard gate," not a #759 dependency. The refusal message this design specifies ("recipient X is not your connected wallet ‚Äî send from Ledger Live, or use the durable-binding path once available") should be revisited once #756 ships a legitimate non-wallet path, and again once the deferred signed-contact leg (D4's follow-on) ships.

**Implementation and verdict sequencing.** DEV implements #757/#760 from this document, not from either incident's own comment thread. The pre-sign exception surface (D8's LiFi decode-coverage claim, D9's residual-scope claims) requires a SEC verdict alongside REVIEW's structural/completeness verdict before DEV starts ‚Äî this is explicitly the kind of design-gates-implementation case where building ahead of the verdict risks a wasted cycle or a shipped non-fix.

**#759 does NOT cover ‚Äî each is a separate mitigation track, its own artifact, its own SEC + REVIEW verdict:**

## 7. Residual risk (recorded, not gating this increment)

Risks named here are known, accepted, or deferred-with-reason ‚Äî recorded so the design's actual coverage boundary is legible, not because any one of them individually blocks #759:

- **`prepare_token_send(to=ATTACKER)` is device-render-only** (D3) ‚Äî the accepted usability trade-off of dropping the hard gate on user-directed transfers; no cryptographic or provenance backstop exists at this seam for that call shape.
- **Uniswap `mint`/`collect.recipient` is user-directed, defended by on-device render only ‚Äî same residual class as `transfer.to`** (D3) ‚Äî `prepare_uniswap_v3_mint`/`_collect` let the user name a fresh recipient, so this bucket-4 argument carries the same accepted residual as `prepare_token_send`: no cryptographic or provenance backstop beyond the device render, for calls that carry no `acknowledgedNonProtocolTarget` stamp.
- **The asset dimension** (D9) ‚Äî this gate authorizes the recipient address, never the asset or its price; a bad-asset/bad-price drain at an honest address is untouched by this design and is the swap-tooling's job, not this seam's.
- **EigenLayer `strategy` and Morpho `marketParams.oracle` sit in the NON_RECIPIENT_ALLOWLIST (D2 bucket 1) on an external, unverified premise** ‚Äî that the named strategy/oracle contract is itself legitimate infrastructure, not attacker-deployed. This design does not independently verify that premise; it is recorded as an external dependency, not gated on a durable-binding check here (repo CLAUDE.md Inv #15, durable-binding to a verified candidate, is the pattern that would close this if adopted for these arguments in a future revision).
- **Multicall breadth** (D7 property 4) ‚Äî the total decoded-sub-call budget bounds cost and recursion, but the specific budget value is an implementation constant this doc does not fix; DEV sets it against real multicall shapes the MCP's own tools produce.
- **Out-of-scope tracks ‚Äî #760-core, #761, #762** ‚Äî correctly out of this design's scope; see ¬ß5's explicit scope boundary for what each covers and the confirmed sequencing.
- **Preview-only until the send-time re-check ships.** Until ¬ß5's send-time re-check lands, the recipient/authority verdict is enforced only at `preview_send`, inheriting send-time coverage solely through the existing gas-pin/`previewToken` binding. This is the residual state prior to ¬ß5's primary enforcement shipping, not an ongoing gap once it does.
