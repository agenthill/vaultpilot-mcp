## 1. Problem & threat model

**Threat actor.** A rogue or prompt-injected agent — in-model for this repo, which treats all agent-supplied input (tool arguments, `prepare_*` call shape, even which tool is called) as attacker-influenced. No user social-engineering is assumed beyond the user's ordinary Ledger confirm; the device screen is the only thing the threat model treats as trustworthy, and only for what it actually renders.

**The structural gap.** `assertTransactionSafe`'s block 5 (`src/signing/pre-sign-check.ts:357-404`) validates **which function is being called** — the 4-byte selector must belong to the destination's curated ABI — and never validates **where the value or authority goes**. For any recognized destination (`classifyDestination` returns non-null), block 4's argument-agnostic catch-all (`if (!dest)`, `pre-sign-check.ts:336`) is skipped entirely, because it only fires on *unrecognized* destinations. Block 5 then passes any selector on the ABI with no `decodeFunctionData`, no argument read, on the recognized-destination path. The result inverts the intended security model: being on the protocol allowlist makes a recipient-bearing call **more** likely to reach signature than an unrecognized one, not less — an unknown destination is refused unless acked; a recognized one with a free `to`/`receiver`/`owner` argument is not gated at all.

**Asset at risk.** Whatever the recognized protocol call moves: Aave collateral (`withdraw.to`), Morpho supplied/borrowed assets (`withdraw`/`borrow`/`withdrawCollateral.receiver`), Uniswap swap output (`exactInputSingle`/`exactOutputSingle.recipient`), Lido withdrawal NFTs (`requestWithdrawals.owner`), and self-funded credit positions (Aave `supply.onBehalfOf`, Morpho `supply`/`supplyCollateral.onBehalf`). (Permit2 and ERC-4626 are NOT currently recognized destinations — struck from this list, see D5/D7's residual and §7 — a call to either resolves `dest === null` today, outside this seam entirely.) No acknowledgement flag anywhere in the chain is load-bearing against this — the schema-required `acknowledgeNonProtocolTarget: true` literal cannot distinguish a benign call from a drain because every call sets it, and per block 4's skip, the pre-sign layer does not even consult it on this path.

## 2. The failure class — one substrate, four incidents

This is not a new bug class; it is the fourth occurrence of the same substrate defect surfacing in a different corner of the recognized-destination surface:

- **#711** (closed → PR #727) — `applyCustomCallClassifier`'s `transferFrom` ack-bypass checked only `args[0]` (the `from`); the recipient `args[1]` (`to`) was never inspected, so an ack-stamped `transferFrom(from=X≠wallet, to=ATTACKER)` rode through unchecked.
- **#737** (sibling, found at #727's review) — the *same* classifier's `transfer(address to, uint256)` path, which moves the **wallet's own** tokens, could still be ack-bypassed to an arbitrary recipient.
- **#741** (untracked sibling, found by REVIEW's post-merge sweep) — ERC-777 `operatorSend(address,address,uint256,bytes,bytes)` has identical drain semantics to `transferFrom` and was outside the classifier's ruleset entirely.
- **#757 / #760** (this design's trigger, found by SEC's 28-agent adversarial fan-out while adjudicating #737) — the same substrate, one layer up: it is not the `custom_call` classifier that misses the recipient, it is `assertTransactionSafe` block 5 itself, for **every** recognized protocol destination, not just `custom_call`-reachable selectors. #757 is selector-membership-without-argument-check; #760 (LiFi Diamond, `allowedAbi: null`) is the same gap with no selector check at all.

The recurring pattern across all four: a gate keys on **which selector/rule matched**, and treats "selector recognized" as sufficient without ever decoding **which argument carries the recipient**. #759 is ARCH's design to close this at its true locus — the pre-sign argument level — rather than patching the next selector that surfaces it.

## 3. Design

The core idea — argument-level recipient/authority authorization at the pre-sign chokepoint — survived every review round unchanged, including Fable's; every requested change lands inside the mechanism, not against the idea. The mechanism was revised across six rounds: v1 was an inclusion map (fails open on any unmapped selector, reproducing #757 in miniature); v2 inverted to a fail-closed exemption model; v3 folded round-2's one new blocker (a contact predicate that admitted unsigned entries) plus mechanical residuals; v4.1 (shipped mislabeled v4 — D-14) folded round-3's authoritative SEC + REVIEW verdict — narrowing the hard gate to wallet-only and deferring the signed-contact leg (now tracked at #773), plus corrections at D1, D2, D2-rot, D3, D7, D8, D9, and §5; v5 folded a second authoritative SEC + REVIEW re-verdict against v4.1 — most substantially, an honest restatement of D1's trust root (it is a peer-advertised account set, not a device binding — A4), a full four-bucket re-run at send time rather than a bare recipient compare (REVIEW N-1), a monotonicity restatement of D1's ack-independence (N2), and closing the `NON_EVM_RECEIVER_SENTINEL` exemption's seam-level gap (N3). **v6 folds SEC's approval-with-conditions, REVIEW's third structural round, and an independent Fable adversarial §4-grounding pass** — all three converge on the same defects, entirely inside §4 and its two immediate feeder sections (D6, D8): the D6 authority rows were written against a deleted decode model and one (`setApprovalForAll`) is unreachable as stated; the D8/LiFi rows had the stamp/unstamped PASS-REFUSE directions backwards; six Uniswap token-identity paths were missing from bucket 1 entirely, a NEW boot-blocker Fable alone found; and four normatively-stated runtime refusals carried no falsifier. What follows is the fully consolidated v6 design — not a base plus a changelog.

### D1 — Locus: `runEvmPreSignGuards`, after a fail-closed WalletConnect account-set match (NOT a device binding — see A4 below)

The check does **not** live inside `assertTransactionSafe`. That function receives only `tx.from`, which is agent-supplied (`buildCustomCall` sets `from: p.wallet`, and `p.wallet` is `args.wallet` off the tool call — nothing pins it to the connected device before `assertTransactionSafe` runs). Resolving a recipient against `tx.from` at that point is resolving against a value the attacker also controls: set `wallet = to = ATTACKER` — nothing at that point has confirmed `tx.from` against anything.

The check instead runs in `runEvmPreSignGuards` (`execution/index.ts`, evidence-pointer L1819 at recon), **after** the existing account-set match (evidence-pointer L1841-1852 in the recon commit — re-resolve by reading the account-match call site relative to `runEvmPreSignGuards`'s own body post-decomposition). At that point `tx.from` has been checked for membership in `getConnectedAccounts()`.

**A4 — honest restatement of what that match actually proves (supersedes "device-verified").** A prior revision described this as validating `tx.from == connected device account`. SEC's re-verdict showed that claim false: the account set is **peer-advertised, not device-derived** — the non-EVM signers re-derive from the device and hard-refuse on mismatch (`solana-usb-signer.ts:305-317`, `tron-usb-signer.ts:286-289`, `btc-usb-signer.ts:692-695`, `ltc-usb-signer.ts:920-923`); the EVM/WalletConnect path has no analogue. **The honest property this match establishes is: `tx.from` is a member of the account set advertised by the most-recently-adopted, possibly-restored, possibly-non-Ledger WalletConnect session.** That is materially weaker than "device-verified," and this design states it plainly rather than repeat the stronger, false claim. It remains SOUND against THIS design's threat model — a rogue or prompt-injected AGENT, which cannot control the WalletConnect peer (the user's own Ledger Live or other wallet app) — because the agent cannot forge peer-advertised account-set membership any more than it could forge a device signature; the peer-vs-device gap is a pre-existing EVM/WC-path weakness, orthogonal to what #757/#760 attack, and is recorded as a residual with a recommended (not required) companion hardening (§7), not a blocker to closing this incident class.

**Precondition — the account-match itself must be fail-closed on ALL of its known fail-open branches (there are three, not two), as a SAME-PR co-merge invariant with the recipient check, bound at BOTH the preview and the send-time locus (§5).**
- **Branch (1) — the "empty account set" branch is FOUR distinct producer states, not one.** `getConnectedAccounts()` returns `[]` via: (a) no session survives a restore attempt (not peer-controlled); (b) a settled session carrying no `eip155` namespace (**peer-controlled**); (c) an `eip155.accounts` array that is itself empty (**peer-controlled**); (d) every entry failing the CAIP-10/`EVM_ADDRESS` filter (**peer-controlled**). A peer that forces (b), (c), or (d) — while still passing the liveness probe — defeats the match exactly as an unpaired session would; refusing only on interpretation (a) ("no paired session") leaves (b)–(d) open. **Meta-falsifier (§4): reverting the guard to today's `accounts.length > 0 && !accounts.includes(from)` must turn ALL FOUR producer-states' tests red — if only (a) goes red, this branch is still under-bound.**
- **Branch (2) — falsy/missing `tx.from`.** The guard is itself gated `if (tx.from && !isDemoMode())` (evidence-pointer `execution/index.ts:1841`) — a falsy or missing `tx.from` skips the match on its own, independent of branch (1), because `from` is optional on `UnsignedTx` and some send paths substitute `getConnectedAccounts()[0]` when it is absent.
- **Branch (3) — `isDemoMode()`, previously unnamed, exempted on a false premise.** A prior revision's rationale for the demo exemption — "simulated, no real signature ever produced" — is **false as a general claim**: demo mode has at least two live, non-intercepted signing escapes that this seam's own tool-gating does not cover — `sign_btc_multisig_psbt` (reaches a real Ledger `app.signPsbt`) and `finalize_btc_psbt(broadcast:true)` (reaches a real broadcast) — neither populates `tx.from` or reaches this EVM seam, so the exemption's CONCLUSION still holds for THIS gate, but not for the reason previously given. **Delete the "no real signature" premise; replace it with the narrow invariant that actually justifies the exemption here:** every EVM/TRON/Solana broadcast funnels through the single `sendTransaction` function behind the single `send_transaction` tool, which IS the sole broadcast tool intercepted in demo mode — so demo mode is a safe exemption for this particular recipient/account-match check, on that narrower ground, not on a blanket "demo never really signs" claim this repo's own BTC paths falsify.

A real-signing, non-demo path hitting branch (1)'s four producers or branch (2) must **refuse**, not silently skip the comparison and fall through to an unvalidated `tx.from`. **This precondition does not ship separately from the recipient check — the two land in the same PR — and BOTH branches (1) and (2) must be bound at the SEND-TIME locus (§5), not only at preview:** a prior revision's §5 restated only the empty-account-set branch there; the falsy-`tx.from` branch needs the identical send-time binding (§4 carries a falsifier per producer state).

**Monotonicity, not independence, is the bypass-proof.** A prior revision stated the recipient check's non-bypassability as the check being "structurally independent of the ack flags — it never reads `acknowledgedNonProtocolTarget`..." — that claim directly **contradicted** D3's bucket-4 provenance condition and §4's own required test row, both of which DO read that exact flag; on the precise #757-shaped exploit those two disagree on (`prepare_custom_call(contract=UniswapNPM, data=collect(recipient=ATTACKER), acknowledgeNonProtocolTarget=true)`), the "independence" reading passes the drain and the "provenance" reading refuses it — both readings are faithful to a document that asserts both. **The correct, load-bearing property is monotonicity: no ack flag may ever make this gate LAXER; the server-set provenance stamp is read ONLY in the strictness-increasing direction (D3 bucket 4 reads it to move an argument FROM the permissive user-directed bucket INTO hard-gated evaluation when the stamp is present — never the reverse).** Falsifier (unchanged): an ack-stamped call to a recognized destination with an unauthorized recipient must still REFUSE (§4).

**Two loci, two roles — stated explicitly, not implied by one heading.** `runEvmPreSignGuards` runs at the PREVIEW locus only (`previewSend`'s sole call site); it enforces the byte-dependent bucket classification (D2/D3/D6/D7/D8) — sound to rely on at send time because the tx bytes are frozen between preview and send (`#710`/`#742`, landed). The SEND-TIME locus (§5) is where the one MUTABLE input — the connected-account set — is re-checked, by re-running that SAME classification with a freshly-read account set substituted in. **Both loci enforce; they enforce different inputs, not different mechanisms** — full specification in §5 (this explicit two-loci statement replaces a prior single-locus heading that read, alongside `sendTransaction` never invoking any guard at all pre-#759, as if only one locus mattered).

**Export requirement.** `classifyDestination` is currently module-private to `pre-sign-check.ts` (defined at line 105). It must be exported so `runEvmPreSignGuards` can classify the destination independently. `dest === null` (unrecognized) defers entirely to D9 — no hard gate; an ack-stamped unknown-destination call escapes this seam by design, unchanged from today.

**Decode source: block 5's accept-set union, RE-DERIVED from `classifyDestination`'s own per-kind switch — this is the ONLY decode source.** Block 5's real accepted-selector set for a given `dest.kind` is, in several cases, a UNION across ABIs — not the single curated `allowedAbi` field alone. Sharply for `weth9`: `dest.allowedAbi` is `erc20Abi`, but block 5's own acceptance test for that kind is the union `ERC20 ∪ wethAbi` selectors (`pre-sign-check.ts:100-102`, consumed at the per-kind selector-set construction ~L362-396). This union diverges from the single curated `allowedAbi` field on four destination kinds (`weth9` and Lido/Rocket Pool's `stETH`/`rETH` unions are concretely traced; DEV re-derives the exact fourth against the live per-kind switch). D2's decode-and-enumerate step, and D2-rot's static enumeration, must both walk the SAME union block 5 actually accepts for a `dest.kind`, re-derived locally from `classifyDestination`'s own switch. **A prior revision offered a second option here — routing through `decodeCalldata` (`signing/decode-calldata.ts:545`) — that option is DELETED, not merely de-emphasized.** `decode-calldata.ts` maintains its OWN, already-diverged `classifyDestination`/`DestinationKind` (10 kinds vs. `pre-sign-check.ts`'s 14 — missing `lido-wstETH`, `rocketpool-depositPool`, `rocketpool-rETH`, and the entire `uniswap-v3-swap-router` kind; `grep -c "swapRouter02\|uniswap-v3-swap-router" src/signing/decode-calldata.ts` → 0). Routing D2/D2-rot through it would silently drop the ENTIRE Uniswap swap surface from decoding: an `exactInputSingle(recipient=ATTACKER)` or a `multicall([…])` to SwapRouter02 resolves `null` under `decodeCalldata`'s own classifier and never reaches D2's bucket walk, D3's hard-gate rows, or D7's multicall gate — while still passing block 5 (which DOES recognize the swap router) — reproducing #757 by a doc-level implementation choice. This is also the exact rot D2-rot's own domain-sourcing rule (below) forbids: a second, independently-maintained classification list. Using `dest.allowedAbi` alone as the decode source (the other thing this rule forbids) silently under-covers the diverging kinds: a legitimate call on a selector present only in the wider union would either throw against the narrower curated ABI (an over-block — D2-rot's top-level-decode-refuse firing on a VALID call) or decode against the wrong shape and misclassify an argument.

**Predicate normalization: `getAddress(...) === getAddress(...)`, not a bare string compare.** The wallet-only predicate (D4) and every hard-gated recipient comparison in this design normalize BOTH operands via viem's `getAddress` (or equivalent checksum-canonicalization) before comparing — `tx.from` is agent-supplied and case-arbitrary (its schema regex-validates hex shape only, not case), and a decoded recipient's case comes from whatever the calldata encoded. The repo already normalizes both sides at every existing comparison site (`execution/index.ts:1843-1844`, `swap/index.ts:214`, `tron/lifi-swap.ts:236`) — this design's predicate matches that convention rather than introducing a bare `===`. No false PASS is possible from skipping this (the address alphabet is confined enough that `toLowerCase`/`getAddress` are injective on it) — the only hazard is one-sided normalization, which fails CLOSED (a legitimate, differently-cased recipient wrongly refused), never open. §4 carries a mixed-case over-block test.

### D2 — Fail-closed authorization gate over every address-typed argument

Replaces v1's inclusion map. For every recognized-destination selector that passes block 5, decode the call and enumerate **every address-typed argument path**, including nested tuples and arrays-of-tuples (e.g. Uniswap's `ExactInputSingleParams.recipient`, Morpho's `MarketParams` tuple, Permit2's `AllowanceTransferDetails[]`). Each such address must fall into exactly one of four buckets:

1. **NON_RECIPIENT_ALLOWLIST** — admission is a **stated test**, not a name-list: an address argument qualifies only if **it cannot cause the caller's value to be spent to a third party's benefit, and cannot confer authority on a third party.** Per-entry evidence (replaces the prior bare name-list):
   - Lido `submit.referral` — a fee-routing address; the referral fee is paid from the protocol's own emission, never redirected from the caller's principal or output.
   - EigenLayer `depositIntoStrategy.strategy` — **a named exemption on an UNVERIFIED premise, not a proven property (v5 correction).** A prior revision admitted this entry by the SAME proven-property test as the others here ("the strategy contract itself, not a beneficiary") — reasoning from what the argument is CALLED, not from a verified property — while §7 correctly recorded the same entry as resting on "an external, unverified premise." An entry cannot both satisfy a proven-property gate and be recorded as unverified; that was an internal contradiction. `strategy` is a fully agent-controlled, schema-validated address (`execution/schemas.ts:897-899`) passed straight into the call (`staking/actions.ts:288`) with no binding to any curated venue list — whether an attacker-named `strategy` can actually drain depends entirely on EigenLayer's OWN deposit whitelist, external behavior with no repo artifact, so this design asserts NO drain claim either way. It is kept in bucket 1 as an explicit **named-exemption-on-an-unverified-premise**, distinct from the proven-property test the other entries here meet. Hard-gating it wallet-only is **NOT** an option — every legitimate `prepare_eigenlayer_deposit` call sets `strategy` to something other than the wallet by definition, so hard-gating would refuse all of them. A durable-binding treatment (repo CLAUDE.md Inv #15, a curated venue list) is the correct eventual fix if EigenLayer's own whitelist ever proves insufficient; not adopted this increment.
   - Every Morpho `marketParams.{loanToken, collateralToken, oracle, irm}` — venue/infrastructure addresses fixed by the market's own identity, never a redirectable beneficiary (D2-rot below; the `oracle`/`irm` pair is the easy-to-miss half of this tuple; `marketParams.oracle` shares EigenLayer `strategy`'s external-unverified-premise caveat above — recorded at §7).
   - Aave `borrow.onBehalfOf` — passes the test by **direction**, not by absence of a third party: `borrow`'s ABI (`aave-pool.ts:42-52`) carries no separate receiver parameter, so the borrowed asset always transfers to the caller (`msg.sender`) regardless of `onBehalfOf`; `onBehalfOf` only designates which account's debt is charged, and charging a third party's account requires that party's own prior on-chain credit-delegation approval. The caller's own value is never spent to a third party's benefit here.
   - **Aave `repay.onBehalfOf` and Morpho `repay.onBehalf` do NOT belong in this bucket — moved to D3's hard-gated list, below.** `repay` is the mirror image of `borrow`: it spends the CALLER's own tokens to reduce a THIRD PARTY's debt. Setting the argument to an attacker's address pays down the attacker's obligation using the victim's funds — a real drain the general test above correctly excludes. A prior revision exempted it in error by pattern-matching on the shared `onBehalfOf` argument name rather than checking the direction of value flow.
   - **Seven further bucket-1 address paths, previously unlisted (v5 addition; corrected count — a prior revision said "eight" but enumerated only seven) — all pass the proven-property test above.** Aave `asset` on all four functions carrying it (`supply`/`withdraw`/`borrow`/`repay` — `aave-pool.ts:22,34,45,58`) — the token being acted on, never a beneficiary. Compound Comet `asset` ×2 (`compound-comet.ts:68,78` — **Comet is a recognized destination [`pre-sign-check.ts:115-123`] that appeared NOWHERE in this design's D2/D3/§4 before this correction**, an omission this list now closes). EigenLayer `token` (`eigenlayer-strategy-manager.ts:25` — the deposited token address, same token-identity reasoning as `asset`). All seven fail CLOSED today under D2-rot's boot enumeration (unclassified path → won't boot), so none was ever a live drain — but DEV either could not boot the server or would have invented seven ad-hoc entries without the per-entry evidence this bucket exists to require. **Three further paths belong in D3, not here, and cost nothing to move:** Morpho `withdraw`/`borrow`/`withdrawCollateral.onBehalf` (`morpho-blue.ts:80,96,139`) are the SAME direction/drain class as Aave/Morpho `supply`/`supplyCollateral.onBehalf`, already hard-gated in D3 — every MCP builder for these three hardcodes `onBehalf` to the wallet, so moving them to D3 is free (see D3).
   - **Six Uniswap token-identity paths, previously unclassified — a NEW boot-blocker without them, found only this round (v6 addition).** NPM `mint.params.token0`/`.token1` (`uniswap-position-manager.ts:60-61`) and SwapRouter02 `exactInputSingle.params.tokenIn`/`.tokenOut` + `exactOutputSingle.params.tokenIn`/`.tokenOut` (`uniswap-swap-router-02.ts:26-27,47-48`) — all six are the tokens being swapped or minted, never a beneficiary, the same token-identity reasoning as the Aave/Comet `asset` and EigenLayer `token` entries above. Every prior revision left these six unclassified. Because they sit on state-mutating functions inside block-5-recognized ABIs (NPM `mint`, SwapRouter02 `exactInputSingle`/`exactOutputSingle` — both ABIs already hard-gated for their OTHER address argument, `recipient`, at D3), D2-rot's boot enumeration would find these six unclassified and REFUSE TO BOOT — bricking every runtime §4 row, correct or not, since nothing runs until the server boots. **Corrected total: this bucket now lists THIRTEEN previously-unlisted address paths, not eight** — the seven above plus these six.
2. **Wallet-only (D4)** — resolves to the connected WalletConnect account set (`tx.from`, per D1 — a peer-advertised match, not a device binding; see D1's A4 restatement). The signed-contact leg is **deferred to #773**; see D4.
3. **Authority designation (D6)** — an authorization-conferring argument (spender/operator), handled by the unified authority policy, not the recipient policy.
4. **USER_DIRECTED_RECIPIENT** — a fourth bucket (below) for arguments the tool's whole purpose is to route to an address the *user*, not the protocol, names — pass-through at this gate, governed by the existing resolver+device-display defense instead, and conditioned on the absence of a server-populated provenance stamp — see D3.

**Any address argument that does not classify into one of these four buckets → REFUSE.** Adding a new ABI thus fails toward a human-clearable refusal, never a silent drain — the opposite failure mode from v1's inclusion map and from #757 itself.

**The fourth bucket, and why it must exist (reconciling D2 with D3).** `erc20Abi.transfer.to` (and the stETH/WETH9 `transfer` unions) is a user-directed recipient by the tool's own design — `prepare_token_send` exists precisely to let the user name a fresh address, and that flow is governed by `resolveRecipient` labeling plus the on-device address render (§D3), not a pre-sign hard refusal. Without a fourth bucket, `transfer.to` fits none of buckets 1-3: bucketing it as a recipient (2) reopens the over-block this design deliberately removes (§D3); leaving it unbucketed fails the anti-rot enumerator below, which throws on any unclassified path. The fourth bucket is what lets the module-load enumerator pass while runtime still treats `transfer.to` as PASS, defended by the resolver/device-render layer — D2 and D3 are explicitly reconciled on this argument, not left in tension.

### D2-rot — Module-load anti-rot enumeration (the completeness mechanism)

A hand-seeded classification map rots the moment a new ABI or a new selector on an existing ABI is recognized without a matching entry — exactly the #757 failure mode, recreated one layer down. The anti-rot mechanism: **at module load**, recursively enumerate every address-typed argument path across every recognized ABI and assert each path is a member of `NON_RECIPIENT_ALLOWLIST ∪ RECIPIENT_MAP ∪ AUTHORITY_MAP ∪ USER_DIRECTED_MAP`. **Throw at startup / fail a unit test red** on any unclassified path — this converts "new recognized ABI silently ungated" into "won't boot," a machine-checkable falsifier rather than a hand-maintained completeness obligation.

**Scope note: the enumerator keys on ABIs; the destination-recognition set keys on CONTRACTS.** These are different domains — `classifyDestination` recognizes a destination by its `to` address (a `CONTRACTS`-keyed lookup) and attaches an ABI to it; the enumerator walks the recognized *ABIs* directly. Keeping the two domains distinct in the design's own vocabulary matters for DEV: an enumerator that silently drifted to iterate a separately-maintained ABI list, rather than the exact set `classifyDestination` attaches per contract, would reopen a rot path this section exists to close.

**The enumerator's ABI iteration domain must be DERIVED from the block-5 accepted-selector UNION per kind — not `classifyDestination`'s raw `allowedAbi` field taken alone, and not a separately hand-maintained list (this complements, not replaces, the scope note above).** `classifyDestination` (`pre-sign-check.ts:104-190`) attaches a per-kind `allowedAbi`, but D1's decode-source rule already establishes that block 5's ACTUAL accept-set is a wider union on four kinds (weth9/stETH/rETH/one further kind) — so a boot-time equality assertion against `allowedAbi` alone would brick boot on day one: those four kinds would show a domain mismatch that is not rot, it is this design's own known, deliberate union. State the domain once, correctly: the enumerator's domain = the SAME per-kind union block 5 accepts, re-derived from block 5's own kind→selector switch (D1) — never `classifyDestination`'s narrower `allowedAbi` field taken alone, and never a second, independently-maintained array. **`lifi-diamond` (`allowedAbi: null`) is explicitly EXCLUDED from this domain and routed to D8** — its exemption is a different, narrower mechanism (D8's receiver-extraction + provenance condition), not this enumerator's concern; including it here would demand classifying an ABI that by design has none. A separately hand-listed enumerator domain remains a rot vector structurally identical to the one this section exists to close — the same failure mode D1's `decodeCalldata`-option deletion (N1) closes at the decode-source level applies equally here at the domain-source level: a future PR adding a kind + ABI to `classifyDestination`/block 5 without updating a SEPARATE enumerator list boots clean and silently ungates the new kind, reopening #757 one level up from where D2-rot was built to close it. **Boot falsifier:** assert the enumerator's ABI domain is set-equal to the per-kind union block 5 actually accepts (excluding `lifi-diamond`, routed to D8); fail boot if they diverge in either direction.

Five scoping/state corrections, all load-bearing for the enumerator to be buildable and complete:

- **Scope to state-mutating function inputs only — a semantics exclusion, not a volume-management shortcut.** Restrict enumeration to `stateMutability ∈ {nonpayable, payable}` function **inputs**; exclude `view`/`pure` functions and all address-typed **outputs** (e.g. Comet `getAssetInfo`'s output, Aave `getUserAccountData`'s output tuple, Uniswap `positions()` outputs, ERC-20 `balanceOf`/`allowance` reads). The reason is not that walking read-only surface is impractically large — it is that a read-only address categorically has no home in ANY of the four buckets (it is not a recipient, not an authority, not user-directed; it is not writable state at all), so including it would fail boot BY CONSTRUCTION on every single one, not as a matter of degree. Scoping to state-mutating inputs is a semantic exclusion of a category that cannot be classified, not a shortcut to keep the list short.
- **Fail-closed on opaque byte containers, not just typed addresses.** The enumerator as stated above only walks statically-typed `address` arguments. A recipient can be smuggled inside a `bytes`/`bytes32` argument on a state-mutating function, invisible to that static walk. Any state-mutating function carrying a `bytes`/`bytes32` parameter is **unclassified-unless-explicitly-exempted** — it must be on an explicit MUST-DECODE / MUST-RECURSE / REQUIRE-EMPTY list (below) or the module fails to boot.
- **A third state — REQUIRE-EMPTY — for opaque bytes params with no available decoder.** Five currently-recognized state-mutating functions carry a `bytes`/`bytes[]` parameter: Morpho `supply`/`repay`/`supplyCollateral.data` (`morpho-blue.ts:65/113/128` — each an arbitrary borrower-callback payload with no decoder available to this design) and the two Uniswap `multicall(bytes[])` entries (one per ABI; D7 below, MUST-RECURSE). Morpho's `data` param is **REQUIRE-EMPTY**: refuse the call unless the value is zero-length. This is stricter than a blanket exemption — it does not claim to have proven the callback payload can never carry a recipient, it simply refuses to accept a non-empty one it cannot decode, closing the risk without asserting more than is known.
- **Universal Router example relabeled FORWARD-LOOKING — it was presented as a live gap and is not one.** Uniswap's Universal Router `execute(bytes commands, bytes[] inputs)` is **not currently a recognized destination** — `classifyDestination` recognizes only the NFT position manager and SwapRouter02 — so a Universal Router call resolves `dest === null` and is handled by block 4's catch-all, never reaching block 5 or this enumerator at all. A prior revision cited it as a live example of the bytes-container escape; it is not one today. It remains a useful **forward-looking** illustration of the general risk class (zero top-level address arguments, every recipient inside an opaque blob) that this MUST-DECODE/REQUIRE-EMPTY discipline must also catch the day Universal Router, or anything shaped like it, becomes a recognized destination. The two Uniswap `multicall(bytes[])` entries (D7) are the real, currently-recognized bytes-container path this design closes today.
- **The top-level decode failure also refuses.** Mirror the existing approve-decode catch (`pre-sign-check.ts:262-266`): a top-level `decodeFunctionData` throw on a recognized-destination call is a REFUSE, the same posture already specified for multicall sub-calls (D7) but stated explicitly at the top level too.

A structural falsifier accompanies the enumerator itself: a test asserting it finds `.recipient` nested inside `exactInputSingle`'s tuple parameter, proving the tuple/array-of-tuple walk actually works, not just that it compiles.

### D3 — Scope of the hard gate: protocol-embedded recipients, not user-directed sends

The **wallet-only** hard gate (non-ack-bypassable — see D4) applies only to **protocol-embedded recipient arguments** — arguments where the `prepare_*` tool sets the value to the wallet address by construction, so any other value is a genuine anomaly, never a legitimate call shape the tool itself would produce:

- Aave `withdraw.to`
- Morpho `withdraw`/`borrow`/`withdrawCollateral.receiver`, and their sibling `.onBehalf` arguments (moved from D2 bucket 1 — same free-hardcode-to-wallet direction as the self-funded-credit row below; `morpho-blue.ts:80,96,139`)
- Uniswap tuple `exactInputSingle`/`exactOutputSingle.recipient` (swap-class — `exactInput`/`exactOutput` are **ABSENT** from `swapRouter02Abi` and not applicable; its complete function set is `exactInputSingle`, `exactOutputSingle`, `multicall`, `unwrapWETH9`, `refundETH` — `uniswap-swap-router-02.ts:19,40,61,68,78`), `unwrapWETH9.recipient` (**not** `sweepToken` — absent from `swapRouter02Abi`, so unreachable by decode; already refused via D7's absent-selector-throw rule, not by being on this list)
- Lido `requestWithdrawals.owner`
- Permit2 `transferFrom.to` and batch `.to[]` (**forward-looking** — Permit2 is not yet a recognized destination; gated on D5 shipping)
- Self-funded-credit arguments: Aave `supply.onBehalfOf`, Morpho `supply`/`supplyCollateral.onBehalf` — these are real drains (an attacker-controlled `onBehalf` deposits the user's funds into the attacker's position, withdrawable later by the attacker) even though the *transfer* direction is inbound, because the position/aToken lands at the named address, not the caller.
- **Aave `repay.onBehalfOf`, Morpho `repay.onBehalf` (moved from D2's bucket 1 — see D2's corrected stated test).** A repay call spends the CALLER's own tokens to reduce a THIRD PARTY's debt; setting the argument to an attacker's address pays down the attacker's obligation using the victim's funds — the mirror-image drain of the self-funded-credit row above, on the repayment side rather than the deposit side.

**Also dropped from the hard gate: Uniswap `mint.recipient` and `collect.recipient` — moved to bucket 4 (USER_DIRECTED_RECIPIENT), not hard-gated.** These are user-choosable on the LIVE standalone tools: `prepare_uniswap_v3_mint` and `prepare_uniswap_v3_collect` (`src/index.ts:4613/4631/4679`) build `recipient = p.recipient ?? p.wallet` (`lp/uniswap-v3/actions.ts:190,827`) — a user legitimately mints an LP position, or collects fees, to a fresh address they name, exactly as with `erc20 transfer.to`. Hard-gating them wallet-only would refuse that legitimate flow; a prior revision's "prepare_* sets recipient=wallet by construction" premise did not hold for these two. They inherit bucket 4's full mechanism, not an unconditional pass: governed by the resolver+device-render defense, AND conditioned on the absence of a server-populated provenance stamp (same as `transfer.to`, above) — a `prepare_custom_call` call to Uniswap's position manager stamping `acknowledgedNonProtocolTarget` on a `mint`/`collect` with a non-wallet recipient does NOT qualify for bucket 4's pass-through and is evaluated as hard-gated instead, closing the same #757-shaped exploit on this protocol. The REBALANCE-multicall path (`lp/uniswap-v3/actions.ts:1120,1175`), which hardcodes `recipient=wallet`, is unaffected — it continues to pass under D7's universal-per-leg rule regardless of which bucket `mint`/`collect.recipient` resolves to, because that flow's recipient is the wallet either way.

**Dropped from the hard gate: ERC-20/native `transfer`/send.** A hard refusal on `prepare_token_send` to a fresh, unsaved address contradicts the tool's entire purpose — sending to an address the user names for the first time is the normal case, and block 1 already lets a **native** send to any address through on the strength of the on-device recipient display alone; refusing the equivalent ERC-20 `transfer` at this gate is an inconsistency the pre-sign layer cannot resolve on calldata alone (that distinguishing signal — which tool built the call, and what the user was shown — lives above this gate).

**A prior revision's basis for this was FALSE and is corrected, not restated.** The prior text asserted "`transfer`-via-`custom_call` is already closed by the #741-family classifier" and that a fresh address is "resolver-labeled." Neither holds. `transfer(address,uint256)` (`0xa9059cbb`) cannot structurally be a #741-family gate: that gate's rule shape requires a `from`/`owner` argument to compare against the wallet (the #711/#737/#741 pattern), which bare `transfer` does not carry. The custom-call classifier's own `transfer` rule (the #652 rule) is itself ack-bypassable via an optional `acknowledgeKnownExfilPattern` flag, not a non-bypassable gate. And a fresh, never-saved address carries no resolver label at all — "resolver-labeled" does not describe the exact case this bucket exists for.

**Corrected fix: bucket 4 (USER_DIRECTED_RECIPIENT) is conditioned on the ABSENCE of a server-populated provenance stamp — stated as "no server-populated recipient block," not on an assumed upstream gate.** A `transfer.to` argument reaching this seam through `prepare_custom_call` carries `acknowledgedNonProtocolTarget`, a flag stamped by the SERVER (not the agent) on the custom-call path (`execution/index.ts`, evidence-pointer L1724 — one of three stamp sites total, alongside the Curve builder's own stamp and one further site; ARCHITECTURE.md §3.2 currently documents two, the third is a correction tracked under the separate #762 Curve mitigation track, not this design). A call carrying that stamp does NOT qualify for bucket 4's pass-through and must instead be evaluated as a protocol-embedded/hard-gated argument, or refused if unclassifiable. A `transfer.to` with **no** such stamp — i.e. built by `prepare_token_send`, which never sets it — qualifies for bucket 4's pass-through.

**U1 SETTLED — `acknowledgedNonProtocolTarget` is NOT agent-settable (v6 correction — SEC verified this; a prior revision flagged it UNVERIFIED and required a DEV trace before adoption).** SEC traced every write and read site for this flag: exactly three server-side writers stamp it (the custom-call path and the Curve builder's own stamp, named above, plus one further site), exactly one reader consumes it, no fourth writer exists anywhere in the codebase, the flag carries no zod input-schema exposure on any tool, and no passthrough/catchall path reaches it. This closes the trace a prior revision required before adopting the provenance-stamp condition — bucket 4's gate is adopted as specified here, not conditionally. **CI assertion (SEC-suggested):** assert no fourth writer of `acknowledgedNonProtocolTarget` ever appears in the codebase — a regression guard against a future write site silently reopening exactly the question just settled.

**Residual, stated honestly — not claimed defended by a gate that doesn't exist.** `prepare_token_send(to=ATTACKER)` is defended ONLY by the on-device address render — there is no cryptographic or provenance-based backstop at this pre-sign seam for that specific call shape. This is the accepted, named residual of dropping the hard gate here (the usability rationale above), not an oversight papered over by a false claim of coverage.

### D4 — WALLET-ONLY hard-gate predicate (signed-contact leg DEFERRED)

**A prior revision's wallet-or-signed-contact predicate is superseded. The hard gate's predicate is now `recipient === tx.from` (the WalletConnect-account-set-verified wallet per D1's A4 restatement — a peer-advertised match, not a device binding) — ONLY.** Round-3 SEC review found the signed-contact leg's trust root unsound at its foundation, not merely under-specified: `verifyEvmBlob` recovers the signature against `blob.anchorAddress` (`contacts/verify.ts:221-226`, `verifyMessage({address: blob.anchorAddress, …})` at `:225`) — a field carried *inside the signer's own payload*, not any independently-known address — and `reverseLookup` calls `verifyEvmBlob` directly, with no call anywhere in `src/contacts/` to `getConnectedAccounts()`. An attacker can self-sign a blob whose `anchorAddress` is the attacker's own choice, producing a clean `state: "match"` verdict with no device involvement at all.

**A prior revision's claim here was false and is removed, not softened:** it asserted "`verifyEvmBlob` recovers the signer against the device's own EVM address." That is not what the code does — it recovers the signer against `blob.anchorAddress`, a self-referential, attacker-choosable field.

**Decision: the hard gate accepts ONLY `recipient === tx.from` (D1's account-set match).** This removes an unsound trust root that had no legitimate exercising path to begin with, at no functional cost: no PROTOCOL-EMBEDDED hard-gated argument (D3's list) is ever set to a contact by any live tool. **[A prior revision's broader claim here was false and is corrected, not restated: "every current protocol builder hardcodes its recipient/receiver/onBehalf argument to the wallet address" is contradicted by this same document's own D3 (Uniswap `mint`/`collect`) and D8 (`prepare_swap`'s `toAddress`) sections, and by a live green test (`test/swap-evm-to-tron.test.ts:436-466`).]** The narrower claim that actually holds, and that is what makes removing the contact leg free: every PROTOCOL-EMBEDDED hard-gated argument specifically is hardcoded to wallet by its builder; the tools that let a user legitimately name a fresh non-wallet recipient are routed to bucket 4, not this hard gate, precisely because their recipient argument is NOT protocol-embedded-by-construction. This is simpler, aligned with the design's own simplify mandate, and unblocks #757/#760 on a trust root that is sound as specified, rather than gating the incident fix on a device-binding invariant that does not yet hold.

**The signed-contact leg is DEFERRED, not dropped outright.** It remains the right direction once its prerequisites land: (1) a real device-anchor binding for the contact blob — the actual fix to the defect above, anchoring the address to something the device confirms independently rather than a self-asserted field inside the signed payload; (2) a single-entry-delta approval model (approving the one changed/added entry, not a whole-blob signature that also re-authorizes every unchanged entry alongside it); (3) a parallel device-binding treatment for the Bitcoin contact-signing path (preimage/sha256 handling). **Tracked at #773.**

**One invariant from the contact-signing surface stays verified true and remains load-bearing even under wallet-only — load-bearing on the SURVIVING bucket-4 path specifically, where `resolveRecipient`/`reverseLookup` (`src/contacts/resolver.ts`) is still consumed (`execution/index.ts:1438,1482`), not on the deleted hard-gate contact leg:** unsigned contact entries cannot be laundered into a signed blob. `addContact`'s signed-path branch builds its `oldEntries` list exclusively from the on-disk **signed** blob (`readContactsFile()` → `file.chains[chain].entries`, `contacts/index.ts:300,321`) — the unsigned, agent-mintable in-memory store (the separate no-Ledger-paired branch, `contacts/index.ts` ~L280-295) is never read by the signed-path code at all, so an unsigned entry cannot ride into a subsequently-signed blob. This does not rescue the contact leg for this increment — it is deferred regardless, on the anchor-binding defect above — but it rules out one additional laundering path SEC checked for and confirmed closed.

**Tamper handling for user-directed sends is unchanged.** For user-directed sends (§D3, not hard-gated), a tampered contact entry stays non-fatal warn-and-proceed — the existing prepare-time behavior; this design does not touch that path.

**Residual, not a #759 regression — the whole-blob signing model itself.** Independent of the anchor-binding defect above (which blocks the contact leg entirely for this increment), the signed device prompt shows the **entire** `VaultPilot-contact-v1:{JSON}` blob (`signers/contacts/evm.ts:5-14`), not a per-address confirmation screen — "signed" means "the device holder approved this whole blob," not "confirmed this specific address." This is a property of #773's design space, not #759's, and motivates one of the three prerequisites named above (the single-entry-delta approval item).

### D5 — Permit2: exact-selector scope, not a broad ABI grant

**Permit2 is NOT currently a recognized destination — this whole section is forward-looking.** No `src/abis/permit2*` exists and `classifyDestination` has no Permit2 branch; a Permit2 call today resolves `dest === null`, outside this seam (D1's export-requirement paragraph). This section specifies the design for adding it. Add Permit2 to the recognized-destination set with an ABI scoped to **transfer selectors only**: `AllowanceTransfer.transferFrom(address,address,uint160,address)` (`0x36c78516`, recipient = `args[1]`) and its batch form (`transferFrom(AllowanceTransferDetails[])`, recipient = each element's `.to`; verify the batch selector hex against viem before landing — still unverified). `approve`/`permit` (authority grants) are **excluded** from this ABI grant — see D6's tiebreak, below — admitting them here would open an authority grant ack-free. **The recognized ABI must be exact-selector, not the full Permit2 interface**: `SignatureTransfer.permitTransferFrom`/its batch form (`transferDetails.to`, tuple/array-nested) and `lockdown` must either be mapped with their own nested `.to` paths or remain outside the recognized ABI so block 5 fail-closes them — do not admit a broader Permit2 ABI than these named selectors.

### D6 — Unified authority-designation policy

The gate covers value/credit recipients **and** authority designations — an address argument that confers spend/transfer authority without itself receiving anything is drain-equivalent and must be covered by the same policy: ERC-20 `approve` spender, ERC-721/1155 `setApprovalForAll` operator, Permit2 `approve`/`permit` spender. This reconciles with block 2's existing `approve` spender-allowlist (`pre-sign-check.ts:~237-280`) — authority designation is **one** policy consumed from both loci (see D10), not two independently-maintained lists that can drift apart.

**D6's ONLY live-today authority coverage is ERC-20 `approve` spender, and it runs through PRE-EXISTING block 2, not a new D6 predicate (v6 correction).** Block 2 (`pre-sign-check.ts:237-301` — the allowlist ∪ ack ∪ zero-revoke logic) already decodes and gates `approve` spender on every recognized token destination; D6 does not add a second wallet-only check on top of it, it names block 2 as the policy this design relies on for `approve` and asks that any FUTURE authority-designation addition (below) be reconciled with it rather than drifting into a parallel list.

**Tiebreak: D5 excludes Permit2 `approve`/`permit` from the recognized ABI; this row is forward-looking, not live.** D5 scopes the Permit2 ABI grant to transfer selectors only and explicitly excludes `approve`/`permit`. Read together with D5 as written, this row is unreachable today: since Permit2 is not yet a recognized destination at all (D5), and D5's own eventual ABI grant excludes `approve`/`permit`, a Permit2 `approve`/`permit` call is refused at block 5 (unrecognized selector on an unrecognized-or-transfer-only-scoped destination) before this policy ever runs. **This D6 row is forward-looking**: it specifies the policy Permit2 `approve`/`permit` must be evaluated under IF a future revision ever widens D5's ABI grant to include them — it does not describe a live-today code path, and D5's exclusion is not superseded by this row's existence.

**`setApprovalForAll` is ALSO forward-looking, not live — a prior revision's "live" label was false (v6 correction).** `setApprovalForAll` (`0xa22cb465`) is not on `UNISWAP_NPM_SELECTORS` — the NPM ABI this design recognizes is `balanceOf`/`ownerOf`/`tokenOfOwnerByIndex`/`positions`/`mint`/`increaseLiquidity`/`decreaseLiquidity`/`collect`/`burn`/`multicall`, with no `setApprovalForAll` entry. A `setApprovalForAll` call to the NPM is therefore refused PRE-DECODE at block 5 (`pre-sign-check.ts:398-403`) on EVERY path, stamped or not — NPM is a recognized destination, so an ack never skips block 5 for it (D1's monotonicity). The prior revision's row was written against the deleted `decodeCalldata` model, where the selector exists only presentationally (`decode-calldata.ts:425`) and in the custom-call classifier's advisory rule (`custom-call-classifier.ts:234`) — neither is this seam. Its stated PASS condition (`operator == wallet`) is **unreachable**: even a wallet-operator call never reaches a decode step to check, because the selector itself is refused first, for a reason that has nothing to do with the operator value. **Fix: relabel `setApprovalForAll` forward-looking too — gated on a curated-ABI addition D6 must FIRST specify (adding the selector to a recognized ABI), which this design does not currently do**, unlike Permit2 (D5 already specifies exactly that kind of addition for Permit2's transfer selectors). Until such an addition is specified and lands, D6's authority coverage for `setApprovalForAll` is forward-looking policy, identical in status to the Permit2 `approve`/`permit` row above — not a second live check this design ships today.

### D7 — Multicall: universal per-leg, bounded, cached

`multicall(bytes[])` (`0xac9650d8`, present on both Uniswap ABIs) requires four properties together, not any one alone:

1. **Universal per-leg invariant.** Decode every sub-call and apply D2 to **every leg**, not an existential "does *some* later leg release to the wallet" check — that shape is launderable: `multicall([exactInputSingle(recipient=router), exactInputSingle(A→B, recipient=ATTACKER), unwrapWETH9(min,wallet)])` satisfies "a later leg releases to wallet" while draining through the unchecked middle leg.

   **Provenance-stamp threading (v5 addition).** "Apply D2 to every leg" means the WHOLE D2 mechanism, including D3's bucket-4 provenance condition — the TX-LEVEL `acknowledgedNonProtocolTarget` stamp (D3) must be threaded down into EVERY per-leg evaluation, not read once and forgotten inside the leg walk. Without this, `prepare_custom_call` reaching Uniswap's NPM with `multicall([decreaseLiquidity, collect(recipient=ATTACKER)])` carries the stamp at the tx level (block 4 never fires, block 5 admits `multicall`); a leg evaluator that reads only each leg's own calldata — with no memory of the tx-level stamp — would classify `collect.recipient` as bucket 4 (user-directed, unstamped) and PASS it: the same #757 shape one level down, inside a multicall. Every per-leg bucket-4 classification inside this walk consults the SAME tx-level stamp D3 defines; a leg has no separate provenance signal of its own to derive.
2. **Router-self exception — conjunctive and asset-bound (was an existential, exploitable disjunction).** A swap-leg `recipient == the exact chain SwapRouter address` is permitted ONLY when BOTH hold: **(a)** that leg's `tokenOut == WETH` (a router-self recipient is legitimate only as the intermediate stop before an unwrap, never for an arbitrary output token), AND **(b)** the SAME multicall contains a sibling `unwrapWETH9(amountMinimum > 0, recipient)` leg whose `recipient` independently passes D2. **`refundETH()` does NOT satisfy requirement (b)** — it takes no arguments and releases native ETH unconditionally to `msg.sender`, so it cannot serve as the D2-checked terminal leg the coexistence requirement demands; a router-self leg paired only with `refundETH()`, with no `unwrapWETH9` present, REFUSES. Without BOTH (a) and (b) present together, a router-self recipient on any leg REFUSES — this closes the prior existential hole, where a non-WETH swap leg with `recipient == router` could ride through on an unrelated `unwrapWETH9`/`refundETH` leg elsewhere in the same call that had nothing to do with that leg's own output token. **The terminal release leg itself is WALLET-ONLY** (consistent with D4 — not wallet-or-contact), and never sanitizes sibling legs regardless of how many legs use the router-self exception.

   **Ordering clause (v5 addition — requirement (b) was index-agnostic and is now positional).** The qualifying `unwrapWETH9` terminal-release leg must appear AFTER the router-self leg it pairs with, in decode order — `multicall([unwrapWETH9(1 wei, wallet), exactInputSingle(recipient=router, tokenOut=WETH)])` (terminal release FIRST, before the router-self leg's proceeds even exist) does NOT satisfy requirement (b), even though (a) and (b) as originally stated read as satisfied by token-match and presence alone. The live builder already emits `[swap, unwrap]` in that order (`uniswap-swap/index.ts:249-260`), so this ordering clause costs nothing against any real flow. **`amountMinimum > 0` bounds nothing economically** — 1 wei satisfies it — and must not be read as an economic guarantee; it is a presence/shape check only (cross-reference D9's asset dimension, which is the section that actually addresses value/amount concerns, not this property). **"Sibling" means: within the SAME top-level `multicall` array, not nested inside a further-decoded sub-call** — this design does not itself specify a semantics for a router-self leg and its terminal-release leg living at different nesting depths (property 4's bounded-recursion caps total depth, but does not define cross-depth "sibling-ness"); DEV implementing nested multicalls must either flatten to one array before applying this rule or explicitly define nested sibling-ness before shipping — two implementers reading this document as written could build two different gates on that exact question.
3. **Fail-closed on unrecognized sub-call selectors.** Re-validate every sub-call selector against block 5's `allowedSelectors` for the destination ABI; a selector absent from that ABI (e.g. SwapRouter02's `sweepToken`/`sweepTokenWithFee`/`unwrapWETH9WithFee`, which are absent from `swapRouter02Abi`) must throw on decode and REFUSE, never silently fall through as unchecked.
4. **Total decoded-sub-call budget (breadth-bound, not depth-bound).** Refuse beyond a small fixed TOTAL count of decoded sub-calls across all recursion depths combined — a shallow-but-wide call (e.g. one `multicall` with many sibling top-level legs) needs the same bound a deep one would, so the budget is on total decoded sub-calls, not depth (bounded-loop canon, never "recurse or refuse unboundedly" on depth alone).

**Contact-blob caching — specified for #773's follow-on, NOT built this increment (v5 correction).** A prior revision embedded, inside property 4 above, a normative build instruction to "resolve the signed-contacts blob once per `assertTransactionSafe`/gate invocation" — that locus reference was ALSO wrong (the gate is explicitly NOT inside `assertTransactionSafe`, D1) and, more importantly, stating it as a normative instruction for THIS increment invites DEV to build blob resolution into #759's live call graph, dragging `verifyEvmBlob` — the exact trust root D4 just deleted — back in as dead code with no caller that needs it. Under D4's wallet-only hard gate, NO contact-blob resolution occurs anywhere in this check today. The caching discipline itself remains correct design: resolve the signed-contacts blob once per gate invocation (in `runEvmPreSignGuards`, not `assertTransactionSafe`) and thread it through every leg's check, rather than re-reading disk and re-running `verifyEvmBlob` once per leg — but this is the design for **#773's** follow-on, when the deferred contact leg ships, not a build instruction for #759. DEV implementing #759 today builds NOTHING from this paragraph; it exists so #773's implementer does not have to re-derive the caching requirement independently.

### D8 — LiFi: gate the extractable receiver, ack-gate (fail-closed) the remainder

LiFi Diamond is `allowedAbi: null` and therefore block-5-exempt (ARCHITECTURE.md §3.4) — but it still carries `_receiver` in bridge/swap calldata that the signing layer already decodes and discards (via the already-imported `lifiDiamondAbi` + `tryDecodeLifiBridgeData`). Enumerate the LiFi route shapes whose `_receiver` those decoders can extract, and classify that value per D2's four buckets, using the SAME provenance-stamp discriminator D3 defines — this is what makes the classification seam-evaluable (decidable from calldata + stamp alone, not from which tool built the call): **UNSTAMPED** ⇒ `_receiver == wallet` is bucket 2 (wallet-only, D4); `_receiver != wallet` is bucket 4 (USER_DIRECTED_RECIPIENT — a user-supplied `toAddress`, N-D8 below; this tool legitimately lets a user route an EVM-destination swap's output to an address other than the wallet, and only `prepare_swap` itself produces unstamped calldata). **STAMPED** (reachable only via `prepare_custom_call`) ⇒ hard-gated regardless of value — REFUSED unless `_receiver == wallet`, the identical mechanism D3 applies to `transfer.to`/`mint`/`collect.recipient`. For any LiFi route shape where receiver extraction is not feasible, that shape is downgraded to **ack-gated and fail-closed** with `_receiver` surfaced to the user — never silently exempted the way it is today. Confirm the generic-swap and bridge decoders cover every route facet `prepare_swap` itself can emit, so no route the MCP's own tool produces is silently downgraded to the ack path for lack of decoder coverage.

**`prepare_swap(toAddress)` is a live functional break under an unconditional wallet-only reading; carve it into bucket 4 (v5 addition).** `test/swap-evm-to-tron.test.ts:436-466` — a currently-GREEN test — calls `prepareSwap({wallet, toChain: "arbitrum", toAddress: <fresh address>})` and asserts it is ACCEPTED; the builder honours it (`expectedReceiver = args.toAddress ?? args.wallet`, then throws on `getAddress(decoded.receiver) !== getAddress(expectedReceiver)` — `swap/index.ts:210-219`). An unconditional wallet-only application of D8's main rule would REFUSE this live, intentional flow. Fix: `_receiver` bound to a user-supplied `toAddress` is bucket 4 — device-render-defended, provenance-stamp-conditioned, the same mechanism as `transfer.to`/`mint`/`collect.recipient` — not bucket 2. §4 carries the over-block negative: `prepare_swap(toChain=EVM, toAddress=fresh)` → PASSES.

**Named exemption — `NON_EVM_RECEIVER_SENTINEL` (non-EVM bridge destinations) — conditioned on provenance, and restated as a prepare-time-only invariant (v5 correction — a prior revision described this as a live signing-path defense, and it is not).** When a LiFi bridge route's destination chain is non-EVM, the bridge calldata's `receiver` field is required to carry a fixed protocol sentinel value, not an EVM address (`swap/index.ts:199-202` — the live code throws if the decoded receiver does not exactly match the sentinel). **This invariant is a PREPARE-TIME builder check, not a live signing-path defense.** The only code enforcing it, `verifyLifiBridgeIntent`, is non-exported with its ONE call site inside `prepareSwap` (`swap/index.ts:137-222`, call at `:994`) — zero references under `src/signing/`. The only signing-side use of the sentinel is presentational (`decode-calldata.ts:250` sets a `receiverIsNonEvm` flag purely to render a human-readable string and throws nothing). Meanwhile LiFi Diamond is block-5-exempt (`allowedAbi: null`), so `prepare_custom_call` reaches it with arbitrary calldata and passes every existing pre-sign block — an agent crafting calldata directly can supply the sentinel value itself (a published source-code constant, `lifi-diamond.ts:139`, not a secret) and receive an explicit PASS at the one seam that would otherwise have refused the shape.

**The fix is D3's own bucket-4 mechanism: condition this exemption on the ABSENCE of the server-populated `acknowledgedNonProtocolTarget` provenance stamp.** Calldata carrying NO such stamp (built by `prepare_swap` itself, which never sets it) is exempted as designed. Calldata CARRYING the stamp (reachable via `prepare_custom_call`) is evaluated HARD-GATED instead and REFUSED if the receiver is not wallet — the identical mechanism D3 already applies to `transfer.to`/`mint`/`collect.recipient`. **This relies on the same provenance-stamp guarantee D3 establishes — U1 is SETTLED (see D3's corrected paragraph): `acknowledgedNonProtocolTarget` is not agent-settable, so this condition is adopted as specified, not conditionally.** **There is no per-destination-chain sentinel to reference — delete that framing.** There is exactly one global literal (`lifi-diamond.ts:139`), compared identically regardless of whether the destination is Solana or TRON, carrying zero destination-chain information itself.

**Falsifiers, restated in seam-evaluable terms (receiver value + provenance stamp — not "destination chain"), plus the missing negative.** An EVM-destination LiFi bridge route with `_receiver != wallet`, **STAMPED** → REFUSES (D8's main rule); the SAME route **UNSTAMPED** → PASSES (bucket 4, device-render-defended — see §4's over-block negatives; a v6 correction of a prior revision that had this backwards, refusing the unstamped case). A non-EVM-destination LiFi bridge route carrying the exact expected sentinel value, UNSTAMPED → PASSES (this exemption). A non-EVM-destination route whose receiver field does NOT match the expected sentinel, UNSTAMPED → REFUSES (the exemption is exact-value, matching the live code's own throw-on-mismatch behavior, not a blanket non-EVM pass). **A `prepare_custom_call` call to `LIFI_DIAMOND` carrying the sentinel value AND the `acknowledgedNonProtocolTarget` stamp → REFUSES** (the provenance condition under direct test — the sentinel alone, without the exemption's provenance precondition, must not pass).

*Worth naming, not required this increment:* the only variant that makes this exemption mean anything about the ACTUAL non-EVM recipient (rather than just the presence of a reproducible constant) is a durable binding (repo CLAUDE.md Inv #15) — bind the user-declared `toChain` + `toAddress` into the handle at prepare time and re-check both at the seam. Today `args.toAddress` is format-validated only (`swap/index.ts:65-99`) and never cross-checked against the signed calldata.

This closes only the recipient dimension of LiFi's exposure. The broader arbitrary-calldata-to-LiFi-Diamond bypass — `prepare_custom_call` reaching the same address with arbitrary calldata and arbitrary native value, no selector check at all — is ARCHITECTURE.md §3.4, ruled **UNSOUND — REQUIRES MITIGATION** by SEC (#760) as a wider, separate mitigation thread. #759 does not close #760; it closes the recipient-visibility gap on the routes LiFi's own decoders can read.

### D9 — Explicit residual scope: what this seam does not see

State plainly, not imply coverage: a top-level-argument seam cannot see **nested** calldata, and an address-authorization gate does not itself evaluate asset quality or price on the path it authorizes. Named gaps stay open, owned by separate mitigation threads or stated as an explicit design boundary:

- **Safe `execTransaction`** (ARCHITECTURE.md §3.5) — the outer call carries `safeTxOrigin: true` and skips block 4, but the inner `SafeTx` body (`to`/`data`, potentially `operation === 1` DELEGATECALL) is never decoded by anything. SEC ruled this UNSOUND — REQUIRES MITIGATION. Tracked at #761 (§5's scope boundary).
- **LiFi `SwapData.callTo`** nested inside a struct, beyond what D8's receiver extraction covers. **Currently UNOWNED — stated honestly, not implied tracked.** §5's scope boundary supplies a tracking number for the Safe gap above (#761) and for LiFi's broader arbitrary-calldata bypass (#760-core), but `SwapData.callTo` specifically is scoped differently from #760-core (D8's own text: "closes only the recipient dimension... a wider, separate mitigation thread") and has no number of its own. A reader should not assume this gap is tracked anywhere until it is filed.
- **The ASSET dimension — not a nested-calldata gap, a structural boundary of what this gate claims to do.** D2's gate validates WHERE an argument points; it says nothing about WHAT is transacted or at what price. `exactInputSingle({tokenOut: ATTACKER_DEPLOYED_TOKEN, recipient: tx.from, amountOutMinimum: 0})` passes this design's checks in full — the recipient is the account-set-verified wallet (D1) — while the wallet receives an attacker-deployed worthless token at an unbounded price. **D2's "never a silent drain" authorizes the ADDRESS, not the ASSET.** Asset/price quality is the swap-class prepare-time tools' job (min-out math, quoter enforcement); `prepare_custom_call` bypasses that protection entirely, exactly as it bypasses every other prepare-time invariant (repo CLAUDE.md "Per-protocol `prepare_*` vs. `prepare_custom_call` cutoff").

The Safe gap is a tracked separate mitigation thread; `SwapData.callTo` is a currently-unowned gap, not a tracked one; the asset dimension is not covered by any thread because it is out of this gate's claimed scope by design, not a gap in its execution. The design artifact states all three explicitly so a reader does not assume the argument-authorization gate reaches everywhere a `to`/`data` pair can appear, or that authorizing a recipient says anything about what that recipient receives.

### D10 — Single-source the recipient/authority map across both loci

The #741-family prepare-time custom-call classifier and this pre-sign gate must consume **one** recipient/authority map (one derived from the other, not two independently maintained). **Note:** the divergence this section originally reconciled — prepare-time wallet-only vs. pre-sign wallet-or-signed-contact — does not exist in this increment, because D4's wallet-only revision makes the pre-sign hard gate wallet-only too; the two loci are, for now, identical on the recipient predicate. Single-sourcing the map remains the design going forward regardless: it is what keeps the two loci's selector/argument lists from drifting apart over time on everything OTHER than the wallet-vs-contact predicate (new ABIs, new authority-designation arguments, the C-list), and it is what will let the deferred signed-contact leg (#773) land in ONE place both loci consume when it ships, rather than reopening exactly the drift risk this section exists to prevent.

## 5. Dependencies & sequencing

**#751 is not a #759 dependency.** `safe-tx-store.ts`'s `lookupSafeTx` (#751) is off-path for this gate — it is consumed only by `safe/actions.ts` and `safe/execute.ts` (`execution/index.ts` never calls it), and D9 already names the Safe inner-body decode as a residual this seam does not cover regardless of #751's state. #759's gate runs in `runEvmPreSignGuards`, consuming the EVM tx-store object that `issueHandles` deep-freezes (`tx-store.ts:178`) — that freeze work (`#710`/`#742`) is **landed**, and `sendTransaction` re-reads the SAME frozen object `preview_send` checked, so the two cannot diverge on the frozen tx's fields. **The bytes-level dependency this design rests on — the tx-store deep-freeze — is already landed and satisfied; no open freeze issue gates #759.**

**The send-time re-check RE-RUNS THE FULL four-bucket D2/D3/D6/D7/D8 classification on the frozen bytes, substituting a freshly-read `getConnectedAccounts()` as the wallet input — it is NOT a bare recipient-vs-accounts compare (v5 correction — REVIEW's N-1).** A prior revision specified the send-time gate as "re-decode the tx object... and re-compare its recipient against a freshly-read `getConnectedAccounts()`." Read literally, that is narrower than the four-bucket mechanism, and admits two different, both-faithful implementations: (a) a bare recipient-vs-accounts compare REFUSES a bucket-4 user-directed send (`prepare_token_send`/mint/collect/`prepare_swap`'s `toAddress` — recipient intentionally ≠ wallet, correctly PASS at preview per D3/D4/D8), reintroducing the exact over-block those sections remove; or (b) if preview is truly "UX-only" and send checks only recipient-vs-accounts, D6's authority designations, D7's multicall per-leg walk, and D8's LiFi receiver are enforced NOWHERE at send. **The correct specification: at send time, re-run the SAME D2/D3/D6/D7/D8 classification the preview locus ran, on the SAME frozen tx bytes, with ONLY the account-set input refreshed** — so bucket 4 still passes, hard-gate buckets still refuse a non-wallet recipient, and D6 (authority)/D7 (multicall)/D8 (LiFi) re-enforce against the fresh account set exactly as they did at preview. This re-run is itself fail-closed on the SAME precondition as D1 — an empty refreshed account set, or a falsy `tx.from`, REFUSES at send (A1-A3, D1). Placement: inside `sendTransaction`, AFTER `consumeHandle` (evidence-pointer `execution/index.ts:2272`) and BEFORE `requestSendTransaction` (evidence-pointer `execution/index.ts:2284`).

**Two loci, two enforced dimensions — soften "preview is not the enforcement gate," don't state it as an absolute.** PREVIEW enforces the byte-dependent bucket classification — sound to rely on at send because the tx bytes are frozen between preview and send (`#710`/`#742`, landed). The SEND-TIME re-run enforces the one MUTABLE input, the account set, by re-running that same classification against a fresh read of it. **Both loci enforce; they enforce different inputs, not different mechanisms.** This reconciles a prior revision's unqualified "the send-time re-check is THE gate": it is not that preview enforces nothing, only that the account-set dimension specifically needed a second, later check, since it is the one input the byte-freeze does not cover. Nothing about the send-time re-run is redundant with preview — preview's frozen-byte classification is exactly WHY the send-time re-run only needs to re-EXECUTE the classification (against a fresh account set) rather than re-derive it conceptually from scratch, since the classification's outcome is a function of both the bytes AND the account set, and only the bytes are frozen. (SEC's own framing — the send-time run is the ONLY place a fresh account-set is checked — stays intact; nothing here deletes it as redundant.)

**Free freeze-invariant (SEC-offered, v6 addition): with the contact leg deferred, the account set is the ONLY mutable input, so every byte-dependent bucket's classification must be IDENTICAL at preview and at send.** This falls directly out of two things already established — the tx-store freeze (D1, `#710`/`#742`) and the contact leg's absence (D4) — nothing else varies between the two loci once both hold. **Falsifier:** for a fixed handle, the byte-dependent classification (D2/D3/D6/D7/D8, evaluated on everything OTHER than the account-set comparison itself) computed at `preview_send` must equal the classification computed at `send_transaction`'s re-run; any divergence means the tx-store freeze is broken, since nothing else could explain the two runs disagreeing on a byte-dependent bucket. This divergence is otherwise invisible — each locus reports only its own verdict, and nothing today cross-checks them against each other — so this falsifier is what makes a freeze regression detectable rather than silently drifting.

**This send-time re-check ships in the SAME PR as the recipient/authority gate itself — it is not a follow-on, and it is not optional (N4).** A prior revision's residual note (since corrected, §7) read as if the recipient/authority verdict could ship enforced only at `preview_send`, with the send-time re-check landing later — that would ship a UX-only preview display with NO enforcement anywhere, strictly WORSE than today. D1's account-match precondition already states its own same-PR co-merge invariant; this send-time re-check is bound by the identical invariant, for the identical reason: the gate is unsound with only one of the two loci built.

Falsifiers: (i) swap the connected-account set between `preview_send` and `send_transaction` on the same handle → `send_transaction` REFUSES; (ii) `preview_send` → `preview_send` (re-preview) → `send_transaction` still evaluates the full re-run AT SEND, never carrying forward an earlier preview's verdict. **Cost framing, corrected (D-9) — a prior claim was wrong on both halves:** "the gas-pin guard already re-runs exactly once per pin today at the same call site" is not accurate — `runEvmPreSignGuards` runs ONLY inside `previewSend` (`:1986`), and only on the non-cached path (`previewSend` early-returns before it on a cached pin, `:1964-1984`); `sendTransaction` (`:2165-2284`) invokes NO guard today at all. The correct framing: `sendTransaction` already performs a per-call `consumeHandle` and pin read at this exact site; one additional re-decode-and-classify there is the same SHAPE of cost as work `sendTransaction` already does at that site — not a claim that an equivalent guard run already exists there today.

**Relation to #756.** #756 (recipient-verified send path) is the legitimate feature this seam's refusal message should point users toward for a genuinely new non-wallet recipient on a protocol-embedded argument — it is the durable-binding alternative to "just widen the hard gate," not a #759 dependency. The refusal message this design specifies ("recipient X is not your connected wallet — send from Ledger Live, or use the durable-binding path once available") should be revisited once #756 ships a legitimate non-wallet path, and again once the deferred signed-contact leg (#773) ships.

## 7. Residual risk (recorded, not gating this increment)

Risks named here are known, accepted, or deferred-with-reason — recorded so the design's actual coverage boundary is legible, not because any one of them individually blocks #759:

- **The trust root is a peer-advertised account set, not a device binding — the LARGEST residual this design carries (D1's A4 restatement).** Even with the fail-closed precondition fully bound (D1's three branches, both loci), the gate proves membership in the account set the most-recently-adopted WalletConnect session advertises — not a live device signature. This does not prove a live device is present (a restored-from-disk session can return a non-empty account list with no device attached), does not prove the peer is Ledger Live specifically (peer-pin verification is warn-only, discarded by both callers), and does not bind the chain ID. It is sound against THIS design's threat model — a rogue agent, which cannot forge peer-advertised account-set membership — but not against a compromised or substituted WalletConnect peer, a materially different threat this design does not target. **Recommended (NOT required) companion hardening, tracked separately from #759:** move `probeSessionLiveness` (already exported) to run BEFORE the account-set re-check rather than after, and make the peer-pin verdict BLOCKING on the signing path rather than warn-only. Neither is required to close #757/#760 — the rogue-agent threat model does not need them — but both would close this peer-vs-device gap for a stronger threat model than this design targets.
- **The signed-contact leg is deferred, not built this increment (#773).** Captured in full at D4 and §6; listed here for symmetry with the other deferred-with-reason items in this section.
- **`prepare_token_send(to=ATTACKER)` is device-render-only** (D3) — the accepted usability trade-off of dropping the hard gate on user-directed transfers; no cryptographic or provenance backstop exists at this seam for that call shape.
- **Uniswap `mint`/`collect.recipient` is user-directed, defended by on-device render only — same residual class as `transfer.to`** (D3) — `prepare_uniswap_v3_mint`/`_collect` let the user name a fresh recipient, so this bucket-4 argument carries the same accepted residual as `prepare_token_send`: no cryptographic or provenance backstop beyond the device render, for calls that carry no `acknowledgedNonProtocolTarget` stamp.
- **`prepare_swap`'s `_receiver`, bound to a user-supplied `toAddress`, is the same residual class** (D8/N-D8) — device-render-only, no cryptographic or provenance backstop, for unstamped calls.
- **ERC-4626 `deposit`/`mint`/`withdraw`/`redeem.receiver` is NOT a recognized destination today, and carries no design section proposing to add it (unlike Permit2, D5).** No `classifyDestination` branch exists; a call resolves `dest === null`, outside this seam entirely (D1). A prior revision listed it in D3's hard-gated list and §4's falsifier table as if already covered — which read as coverage that does not exist. If ERC-4626 support is added later, it needs its own D5-equivalent scoping design before landing in this gate's hard-gated list.
- **The asset dimension** (D9) — this gate authorizes the recipient address, never the asset or its price; a bad-asset/bad-price drain at an honest address is untouched by this design and is the swap-tooling's job, not this seam's.
- **EigenLayer `strategy` and Morpho `marketParams.oracle` sit in the NON_RECIPIENT_ALLOWLIST (D2 bucket 1) on an external, unverified premise, stated as a named exemption, not a proven property** (D2) — that the named strategy/oracle contract is itself legitimate infrastructure, not attacker-deployed. This design does not independently verify that premise; it is recorded as an external dependency, not gated on a durable-binding check here (repo CLAUDE.md Inv #15, durable-binding to a verified candidate, is the pattern that would close this if adopted for these arguments in a future revision).
- **Multicall breadth** (D7 property 4) — the total decoded-sub-call budget bounds cost and recursion, but the specific budget value is an implementation constant this doc does not fix; DEV sets it against real multicall shapes the MCP's own tools produce.
- **Out-of-scope tracks — #760-core, #761, #762** — correctly out of this design's scope; see §5's explicit scope boundary for what each covers and the confirmed sequencing.

**[v5 removal]** A prior revision recorded "preview-only until the send-time re-check ships" as a residual here, implying the send-time re-check could legitimately ship as a follow-on. §5's N4 fix corrects this: the send-time re-check is a same-PR co-merge invariant, not a residual state this design tolerates shipping without — there is no valid state in which #759 ships enforced only at preview, so this is no longer listed as an accepted residual.
